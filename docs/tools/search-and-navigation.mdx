## Two Search Dimensions

| Dimension | Tool | Underlying Implementation | Use Case |
|-----------|------|---------------------------|----------|
| **Find files by name** | Glob | ripgrep `--files` + glob filter | "Find all test files", "find files starting with config" |
| **Find code by content** | Grep | ripgrep regex search | "Where is this function defined", "who calls this API" |

Both share the same ripgrep engine, achieving different search modes through different parameter combinations.

## How ripgrep Is Embedded

Claude Code doesn't depend on a system-installed ripgrep — it implements a three-tier fallback strategy in `src/utils/ripgrep.ts`:

```
Priority 1: System ripgrep (USE_BUILTIN_RIPGREP=false)
  -> Uses rg binary from PATH
  -> Security consideration: uses only the command name 'rg', not the full path, to prevent PATH hijacking

Priority 2: Embedded mode (bundled/native build)
  -> process.execPath itself, argv0='rg'
  -> Bun statically compiles rg into the binary, dispatched via argv0

Priority 3: vendor directory (npm build)
  -> vendor/ripgrep/{arch}-{platform}/rg
  -> macOS requires codesign signature + quarantine xattr removal
```

Platform adaptation example:
```
vendor/ripgrep/
  |-- x86_64-darwin/rg      # macOS Intel
  |-- arm64-darwin/rg        # macOS Apple Silicon
  |-- x86_64-linux/rg        # Linux Intel
  |-- arm64-linux/rg         # Linux ARM
  +-- x86_64-win32/rg.exe    # Windows
```

### macOS Code Signing

The vendor-mode rg binary requires ad-hoc signing to pass Gatekeeper (`codesignRipgrepIfNecessary()`):

```typescript
// Executed on first use:
// 1. Check if already validly signed
codesign -vv -d <rg-path>
// 2. If only linker-signed, re-sign
codesign --sign - --force --preserve-metadata=entitlements,requirements,flags,runtime <rg-path>
// 3. Remove quarantine attribute
xattr -d com.apple.quarantine <rg-path>
```

## Search Result Design Considerations

### head_limit & Token Budget

Search results in large projects can have hundreds of thousands of matches. The default returns a maximum of 250 matches — this is not an arbitrary choice but a **token budget** constraint:

- Each matching line is approximately 50-100 tokens
- 250 matches ~ 12,500-25,000 tokens
- This is roughly 6-12% of the 200k context window
- Exceeding this ratio degrades AI reasoning quality

The Grep tool's `head_limit` parameter lets AI adjust as needed — smaller projects can use larger values.

### Sorted by Modification Time

Glob defaults to sorting **most recently modified files first**. This is not the default file system order but a deliberate design decision:

```
Design assumption: Most recently modified files are most likely relevant to the current task
Actual effect: AI sees "active" code first, rather than dormant historical files
```

In `src/tools/GlobTool/`, ripgrep's output is sorted by mtime before being returned to the AI.

### ripgrep Error Handling

ripgrep execution has a dedicated error recovery chain (`src/utils/ripgrep.ts`):

| Error | Handling |
|-------|----------|
| **EAGAIN** (resource exhaustion) | Automatically retries in single-thread mode with `-j 1` |
| **Timeout** (default 20s, WSL 60s) | Returns partial results collected so far, discards potentially incomplete last line |
| **Buffer overflow** | Truncates to 20MB, returns collected results |
| **SIGTERM ineffective** | Escalates to SIGKILL after 5 seconds |

## ToolSearch: Finding Targets Among 50+ Tools

When available tools exceed 50 (including external tools from MCP), AI might not know which one to use. **ToolSearch** (`src/tools/ToolSearchTool/`) provides a tool discovery mechanism.

### Search Algorithm

ToolSearch implements weighted keyword-based search (`searchToolsWithKeywords()`):

```
Input: query = "database connection"
     |
1. Exact match: check if any tool name matches exactly (fast path)
2. MCP prefix match: "mcp__postgres" -> match all postgres-related tools
3. Keyword split: ["database", "connection"]
4. Tool name parsing:
   - MCP tools: "mcp__server__action" -> ["server", "action"]
   - Regular tools: "FileEditTool" -> ["file", "edit", "tool"]
5. Weighted scoring:
   - Tool name exact match: 10 points (MCP: 12 points)
   - Tool name partial match: 5 points (MCP: 6 points)
   - searchHint match: 4 points
   - Description match: 2 points
6. Required word filter: "+database" prefix means must-contain
7. Sort by score, return top-N
```

### `select:` Direct Selection

AI can also use `select:ToolName` to precisely select a known tool. This is faster than searching and supports comma-separated batch selection (`select:A,B,C`).

### Deferred Loading (Deferred Tools)

Not all tools reside in memory permanently. MCP tools and low-frequency tools are marked as `isDeferredTool`, and are only truly loaded after being selected by ToolSearch. This reduces the token overhead of each API call (tool descriptions consume significant tokens).

### Caching Strategy

Tool description retrieval is memoized — the cache is cleared only when the deferred tool set changes:

```typescript
// Tool names sorted and concatenated as cache key
function getDeferredToolsCacheKey(deferredTools: Tools): string {
  return deferredTools.map(t => t.name).sort().join(',')
}
```

## Web Search & Fetching

AI's information retrieval is not limited to local code:

- **WebSearch** (`src/tools/WebSearchTool/`): Calls the Anthropic API's `web_search_20250305` server tool to search the internet
- **WebFetch** (`src/tools/WebFetchTool/`): Fetches specific URL content and converts it to Markdown for AI reading

This lets AI look up documentation, search Stack Overflow, and read GitHub issues — just like human developers work.

### WebSearch Implementation

WebSearch supports two search backends through an adapter pattern, selected by the factory function `createAdapter()` in `src/tools/WebSearchTool/adapters/`:

```
Adapter architecture:
  WebSearchTool.call()
    -> createAdapter() selects backend
      |-- ApiSearchAdapter — Anthropic API server-side search (requires official API key)
      +-- BingSearchAdapter — Direct Bing search page scraping and parsing (no API key needed)
    -> adapter.search(query, options)
    -> Converts to unified SearchResult[] format and returns
```

#### Adapter Selection Logic

The factory function in `adapters/index.ts` selects the backend by the following priority:

| Priority | Condition | Adapter |
|----------|-----------|---------|
| 1 | Environment variable `WEB_SEARCH_ADAPTER=api` | `ApiSearchAdapter` |
| 2 | Environment variable `WEB_SEARCH_ADAPTER=bing` | `BingSearchAdapter` |
| 3 | API Base URL points to Anthropic official | `ApiSearchAdapter` |
| 4 | Third-party proxy / unofficial endpoint | `BingSearchAdapter` |

Adapters are stateless and cached for reuse within the same session.

#### ApiSearchAdapter — API Server-Side Search

Delegates search requests to the Anthropic API's `web_search_20250305` server tool:

```
Call chain:
  ApiSearchAdapter.search(query, options)
    -> queryModelWithStreaming() initiates an independent API call
      -> Carries extraToolSchemas: [BetaWebSearchTool20250305]
      -> API server executes search, returns streaming events
        -> server_tool_use / web_search_tool_result / text alternate in response
    -> extractSearchResults() extracts SearchResult[] from content blocks
```

| Feature | Implementation |
|---------|----------------|
| **Model selection** | Feature flag `tengu_plum_vx3` controls whether to use Haiku (forced tool_choice) or the main model |
| **Search limit** | Maximum 8 searches per call (`max_uses: 8`) |
| **Domain filtering** | Supports `allowedDomains` / `blockedDomains` |
| **Progress tracking** | Streaming parsing of `input_json_delta` to extract query, real-time `onProgress` callback |

#### BingSearchAdapter — Bing Search Page Parsing

Directly fetches Bing search HTML and extracts results with regex, no API key needed:

```
Call chain:
  BingSearchAdapter.search(query, options)
    -> axios.get(bing.com/search?q=...)  — uses browser-level headers to bypass anti-scraping
    -> extractBingResults(html)
      -> Regex matches <li class="b_algo"> blocks
      -> Extracts <h2><a> title and URL
      -> resolveBingUrl() decodes Bing redirect links
      -> extractSnippet() three-tier fallback snippet extraction
    -> Client-side domain filtering (allowedDomains / blockedDomains)
    -> Returns SearchResult[]
```

**Anti-scraping strategy**: Bing returns empty pages requiring JS rendering for non-browser UAs. The adapter uses complete Edge browser request headers (including `Sec-Ch-Ua`, `Sec-Fetch-*`, and other modern browser headers) to ensure full HTML is received. It also uses the `setmkt=en-US` parameter to unify market targeting, preventing Bing from doing region-based redirection based on user IP (e.g., redirecting to German/Singapore markets resulting in irrelevant results).

**URL decoding**: URLs in Bing search results are in redirect format (`bing.com/ck/a?...&u=a1aHR0cHM6Ly9...`), and `resolveBingUrl()` base64-decodes the real target URL from the `u` parameter (`a1` prefix = https, `a0` = http).

**Snippet extraction** (`extractSnippet()`) tries three sources by priority:
1. `<p class="b_lineclamp...">` — snippet paragraph with line truncation
2. `<div class="b_caption">` inner `<p>` — regular snippet paragraph
3. `<div class="b_caption">` direct text content — fallback

| Feature | Implementation |
|---------|----------------|
| **Timeout** | 30 seconds (`FETCH_TIMEOUT_MS`) |
| **Domain filtering** | Supports `allowedDomains` / `blockedDomains`, including subdomain matching |
| **Progress tracking** | Sends query_update and search_results_received callbacks |
| **Abort support** | External AbortSignal propagated to axios request |

### WebSearchTool Unified Interface

`WebSearchTool` (`src/tools/WebSearchTool/WebSearchTool.ts`) is the tool definition facing the main loop; all providers can use it (`isEnabled()` always returns true). It converts the adapter-returned `SearchResult[]` to the internal `Output` format; `mapToolResultToToolResultBlockParam` formats search results as text with markdown hyperlinks and appends a "REMINDER" asking the main model to include Sources in its response.

### WebFetch Implementation

WebFetch is a complete HTTP client + content processing pipeline:

```
Call chain:
  WebFetchTool.call({ url, prompt })
    -> getURLMarkdownContent(url)
      -> validateURL() — length<=2000, no username/password, public domain
      -> URL_CACHE hit check (15-minute TTL LRU, 50MB limit)
      -> checkDomainBlocklist() — calls api.anthropic.com/api/web/domain_info for pre-check
      -> getWithPermittedRedirects() — axios request with custom redirect handling
        -> HTML -> Turndown converts to Markdown (lazy-loaded singleton, ~1.4MB)
        -> Non-HTML -> raw text
        -> Binary (PDF etc.) -> persistBinaryContent() saves to disk
    -> applyPromptToMarkdown()
      -> Truncates to 100K characters
      -> queryHaiku() uses a small model to extract information per prompt
    -> Returns processed result
```

Multi-layer security design:

| Layer | Mechanism | Description |
|-------|-----------|-------------|
| **Domain pre-check** | `checkDomainBlocklist()` | Calls `api.anthropic.com/api/web/domain_info?domain=...`, 5-minute cache |
| **Redirect control** | `isPermittedRedirect()` | Only allows same-host (+/- www) redirects; cross-domain redirects return a prompt for AI to re-call |
| **Redirect depth** | `MAX_REDIRECTS = 10` | Prevents redirect loops from hanging indefinitely |
| **Content size** | `MAX_HTTP_CONTENT_LENGTH = 10MB` | Single response limit |
| **Request timeout** | `FETCH_TIMEOUT_MS = 60s` | Main request timeout; domain pre-check 10s |
| **URL validation** | `validateURL()` | Length, protocol, username/password, public domain checks |
| **Egress detection** | `X-Proxy-Error: blocked-by-allowlist` | Detects enterprise proxy interception |

Pre-approved domains (`src/tools/WebFetchTool/preapproved.ts`):

A domain list that users can fetch without manual authorization, containing ~90 mainstream technical documentation sites (MDN, Python docs, React docs, AWS docs, etc.). The list is divided into hostname-only and path-prefix categories, with O(1) lookup complexity.

For pre-approved domains, WebFetch skips the Haiku summarization step (if content is Markdown and < 100K characters), returning the original text directly — because the structure of technical documentation is already good enough.

Regarding the permission model, WebFetch generates `domain:xxx` rules based on hostname to match user allow/deny/ask rules, supporting permanent allow or deny configuration for specific domains.

### ripgrep Streaming Output

For interactive scenarios (such as QuickOpen), ripgrep supports **streaming output** (`ripGrepStream()`):

```
rg --files -> chunks arrive -> split by line -> onLines(lines) callback
```

No need to wait for ripgrep to finish the entire search — the first batch of results is displayed while rg is still traversing the directory tree. Callers can abort the search early via AbortSignal (e.g., after finding enough results).
