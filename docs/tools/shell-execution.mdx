## Execution Chain Overview

The complete path of a Bash command from AI decision to actual execution:

```
AI generates tool_use: { command: "npm test" }
  |
BashTool.validateInput()         <- Basic input validation
  |
BashTool.checkPermissions()      <- Permission check (see security system chapter)
  |-- isReadOnly()? -> auto allow (read-only commands skip approval)
  |-- bashToolHasPermission()    <- AST parsing + semantic check + rule matching
  +-- No match -> confirmation dialog
  |
BashTool.call() -> runShellCommand()
  |
shouldUseSandbox(input)          <- Whether to wrap in sandbox
  |
Shell.exec(command, { shouldUseSandbox, shouldAutoBackground })
  |
spawn(wrapped_command)           <- Actual process creation
```

## Read-Only Command Detection: Why Read Skips Approval but Bash May Not

BashTool's `isReadOnly()` method (`BashTool.tsx:437`) determines whether a command is considered "read-only":

```typescript
isReadOnly(input) {
  const compoundCommandHasCd = commandHasAnyCd(input.command)
  const result = checkReadOnlyConstraints(input, compoundCommandHasCd)
  return result.behavior === 'allow'
}
```

The detection logic is based on 4 command sets (`BashTool.tsx:60-78`):

| Set | Commands | Nature |
|-----|----------|--------|
| `BASH_SEARCH_COMMANDS` | find, grep, rg, ag, ack, locate, which, whereis | Search commands |
| `BASH_READ_COMMANDS` | cat, head, tail, wc, stat, file, jq, awk, sort, uniq... | Read/analysis commands |
| `BASH_LIST_COMMANDS` | ls, tree, du | Listing commands |
| `BASH_SEMANTIC_NEUTRAL_COMMANDS` | echo, printf, true, false, : | Semantically neutral (don't affect detection) |

For compound commands (`ls dir && echo "---" && ls dir2`), the system splits and checks each segment — **all non-neutral segments must belong to the above sets** for the entire command to be considered read-only.

```typescript
// BashTool.tsx:95 — simplified detection logic
for (const part of partsWithOperators) {
  if (BASH_SEMANTIC_NEUTRAL_COMMANDS.has(baseCommand)) continue  // Skip neutral segments
  if (!isPartSearch && !isPartRead && !isPartList) {
    return { isSearch: false, isRead: false, isList: false }  // Any failing segment -> not read-only
  }
}
```

## AST Security Parsing: tree-sitter Bash Parsing

`preparePermissionMatcher()` (`BashTool.tsx:445`) uses `parseForSecurity()` to parse command structure before permission checks:

```typescript
async preparePermissionMatcher({ command }) {
  const parsed = await parseForSecurity(command)
  if (parsed.kind !== 'simple') {
    return () => true  // Parse failure -> fail-safe, trigger all hooks
  }
  // Extract subcommand list, strip VAR=val prefixes
  const subcommands = parsed.commands.map(c => c.argv.join(' '))
  return pattern => {
    return subcommands.some(cmd => matchWildcardPattern(pattern, cmd))
  }
}
```

Key security point: for compound commands like `ls && git push`, parsing splits them into `["ls", "git push"]`, ensuring `git push` doesn't bypass permission checks because the first half is a read-only command. On parse failure, a fail-safe strategy is adopted — assumed unsafe, triggering all security hooks.

## Timeout Control: Tiered Strategy

```
User-specified timeout -> use directly
  | not specified
getDefaultTimeoutMs()
  |-- Default limit: 120,000ms (2 minutes)
  +-- Maximum limit: 600,000ms (10 minutes, when explicitly set by user)
```

After timeout, the system doesn't kill the process directly — `ShellCommand` (`src/utils/ShellCommand.ts:129`) notifies the caller via `onTimeout` callback, letting the caller decide whether to terminate or background the process.

## Automatic Backgrounding

Long-running commands can be automatically converted to background tasks without blocking the AI's agentic loop:

```typescript
// BashTool.tsx:880
const shouldAutoBackground = !isBackgroundTasksDisabled 
  && isAutobackgroundingAllowed(command)
```

Complete automatic backgrounding chain:

```
Command starts executing
  | progress polling
Not completed within 15 seconds (ASSISTANT_BLOCKING_BUDGET_MS)
  |
Check isAutobackgroundingAllowed(command)
  | allowed
Convert foreground task to background task (backgroundExistingForegroundTask)
  |
shellCommand.onTimeout -> spawnBackgroundTask()
  |
Return taskId to AI, AI can continue doing other things
  |
Background task reports results via notification mechanism when complete
```

The main thread Agent has a 15-second blocking budget — beyond this, the system automatically backgrounds the command. This prevents an `npm install` from blocking the entire agentic loop for several minutes.

## Output Truncation Strategy

Excessively long command output triggers truncation to prevent flooding the AI's context window with massive logs:

| Truncation point | Location | Behavior |
|------------------|----------|----------|
| `maxResultSizeChars` | Tool level (typically 100K chars) | Excessively long output truncated before writing to message |
| Progress poll truncation | `onProgress` callback | Only passes last few lines as progress display |
| `totalBytes` marker | `isIncomplete` parameter | Informs AI that output was truncated |

Truncation is not simply tail-cutting — the `isIncomplete` marker ensures the AI knows the output is incomplete, so it can decide whether to re-fetch with a more precise command.

## Why Use Dedicated Tools Instead of Calling Shell Directly

Claude Code provides dedicated tools (Read, Grep, Glob) for file reading, code searching, and similar operations, rather than having AI use shell commands like `cat` and `grep`. This is not just a UX choice but an architectural design decision:

| Dimension | Dedicated Tool | Bash Command |
|-----------|---------------|--------------|
| **Permission granularity** | `Read` is a read-only operation -> auto-approved | `Bash: cat file` requires approval for the entire command (cat is in the read-only set but takes a different path) |
| **Structured output** | Returns structured data; UI can render diffs, syntax highlighting | Plain text output, no rendering optimization |
| **Performance optimization** | File caching, pagination, token budget control | New process each time, no caching |
| **Concurrency safety** | `isConcurrencySafe()` returns `true` -> can execute in parallel | Bash commands may have side effects, executed serially |
| **Security auditing** | Tool name precisely matches permission rules | Requires AST parsing of command structure before matching |

`isConcurrencySafe()` (`BashTool.tsx:434`) is an often-overlooked but important design — only read-only commands can execute in parallel within the agentic loop; commands with side effects must run serially to prevent race conditions.

## Streaming Progress Feedback Design

BashTool's command execution is streaming, pushing output line by line via `onProgress` callback:

```
runShellCommand()
  |-- Shell.exec() starts child process
  |-- Polls output file every second
  |-- onProgress(lastLines, allLines, totalLines, totalBytes, isIncomplete)
  |   |-- Updates lastProgressOutput / fullOutput
  |   +-- resolveProgress() -> wakes generator yield
  |-- yield { type: 'progress', output, fullOutput, elapsedTimeSeconds }
  +-- return { code, stdout, interrupted, ... }
```

The UI layer displays command output in real time via the `useToolCallProgress` hook. The `resolveProgress()` signal mechanism makes the generator yield only when new data is available, avoiding busy-waiting.
