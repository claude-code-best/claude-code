## Responsibility Separation of the Three Tools

Claude Code splits file operations into three independent tools — this is not a functional division but **risk tiering**:

| Tool | Permission Level | Core Method | Key Property |
|------|-----------------|-------------|--------------|
| **Read** | Read-only (no approval needed) | `isReadOnly() -> true` | `maxResultSizeChars: Infinity` |
| **Edit** | Write (requires confirmation) | `checkWritePermissionForTool()` | `maxResultSizeChars: 100,000` |
| **Write** | Write (requires confirmation) | `checkWritePermissionForTool()` | `maxResultSizeChars: 100,000` |

<Tip>
Read's `maxResultSizeChars` is `Infinity`, but this doesn't mean unlimited output — the actual truncation occurs in `validateContentTokens()` through dynamic token budget-based determination, not a hard character count limit.
</Tip>

## FileRead: Multi-Modal File Reading Engine

Source path: `src/tools/FileReadTool/FileReadTool.ts`

### Read Deduplication Mechanism

The Read tool has an often-overlooked but critical **deduplication layer**. When AI repeatedly reads the same range of the same file, the system doesn't waste tokens sending two complete copies of the content:

```typescript
// FileReadTool.ts:530-573 — deduplication logic
const existingState = readFileState.get(fullFilePath)
if (existingState && !existingState.isPartialView && existingState.offset !== undefined) {
  const rangeMatch = existingState.offset === offset && existingState.limit === limit
  if (rangeMatch) {
    const mtimeMs = await getFileModificationTimeAsync(fullFilePath)
    if (mtimeMs === existingState.timestamp) {
      return { data: { type: 'file_unchanged', file: { filePath: file_path } } }
    }
  }
}
```

Key design points:
- Deduplication only applies to **reads from the Read tool itself** (determined by `offset !== undefined`)
- Edit/Write also write to `readFileState`, but their `offset` is `undefined`, so they don't falsely trigger deduplication
- File hasn't been externally modified is verified via mtime comparison
- Has a GrowthBook killswitch (`tengu_read_dedup_killswitch`) for emergency disabling

Measured data: BQ proxy shows approximately 18% of Read calls hit the same file, accounting for 2.64% of fleet `cache_creation`.

### Multi-Format Dispatch: Four Paths for Text, Images, PDF, and Notebooks

The Read tool's `callInner()` dispatches to four completely different processing paths based on `ext`:

```
.ipynb  -> readNotebook() -> JSON cell parsing -> token validation
.png/.jpg/.gif/.webp -> readImageWithTokenBudget() -> compression + downsampling
.pdf -> extractPDFPages() / readPDF() -> page-level extraction
Other -> readFileInRange() -> paginated reading
```

The **image path compression strategy** is particularly granular:
1. First apply standard scaling with `maybeResizeAndDownsampleImageBuffer()`
2. Estimate token count with `base64.length * 0.125`
3. When exceeding budget, call `compressImageBufferWithTokenLimit()` for aggressive compression
4. If still over limit, use sharp as final fallback: `resize(400,400).jpeg({quality:20})`

The **PDF path** has a page count threshold: when exceeding `PDF_AT_MENTION_INLINE_THRESHOLD` (default value in `apiLimits.ts`), paginated reading is enforced with a maximum of `PDF_MAX_PAGES_PER_READ` pages per request.

### Security Safeguards

The Read tool sets up multiple security gates in `validateInput()`:

1. **Device file blocking** (`BLOCKED_DEVICE_PATHS`): `/dev/zero`, `/dev/random`, `/dev/tty`, etc. — prevents infinite output or blocking hangs
2. **Binary file rejection** (`hasBinaryExtension`): After excluding PDF and image extensions, blocks reading `.exe`, `.so`, and other binary files
3. **UNC path skipping**: Skips file system operations for `\\server\share` paths on Windows, preventing SMB NTLM credential leaks
4. **Permission deny rules** (`matchingRuleForInput`): Directly rejected after matching `deny` rules

### Smart Suggestions When File Not Found

When a file doesn't exist, Read doesn't just report "file not found":

```typescript
// FileReadTool.ts:639-647
const similarFilename = findSimilarFile(fullFilePath)      // Similar extension
const cwdSuggestion = await suggestPathUnderCwd(fullFilePath) // cwd relative path suggestion
// macOS screenshot special handling: thin space (U+202F) vs regular space
const altPath = getAlternateScreenshotPath(fullFilePath)
```

Special handling for the thin space (U+202F) before AM/PM in macOS screenshot filenames — this is a cross-macOS-version compatibility issue discovered through testing.

## FileEdit: Precise String Replacement Engine

Source path: `src/tools/FileEditTool/FileEditTool.ts` + `utils.ts`

### Quote Normalization: What About Characters AI Can't Output

AI models can only output straight quotes (`'` `"`), but source code may use curly quotes (`'` `'` `"` `"`). The `findActualString()` function handles this mismatch:

```typescript
// utils.ts:73-93
export function findActualString(fileContent: string, searchString: string): string | null {
  if (fileContent.includes(searchString)) return searchString      // Exact match
  const normalizedSearch = normalizeQuotes(searchString)           // Curly quotes -> straight quotes
  const normalizedFile = normalizeQuotes(fileContent)
  const idx = normalizedFile.indexOf(normalizedSearch)
  if (idx !== -1) return fileContent.substring(idx, idx + searchString.length)
  return null
}
```

After matching, there's also **reverse quote preservation** (`preserveQuoteStyle`): if the file uses curly quotes, the new replacement string is automatically converted to curly quotes, including apostrophes in contractions (e.g., "don't").

### Atomic Read-Modify-Write

The Edit tool's `call()` method implements a **lock-free atomic update** protocol:

```
1. await fs.mkdir(dir)            <- Ensure directory exists (async, outside critical section)
2. await fileHistoryTrackEdit()   <- Back up old content (async, outside critical section)
3. readFileSyncWithMetadata()     <- Synchronous read of current file content (critical section starts)
4. getFileModificationTime()      <- mtime verification
5. findActualString()             <- Quote-normalized matching
6. getPatchForEdit()              <- Compute diff
7. writeTextContent()             <- Write to disk
8. readFileState.set()            <- Update cache (critical section ends)
```

Steps 3-8 **allow no async operations** (source comments explicitly state: "Please avoid async operations between here and writing to disk to preserve atomicity"). This ensures no other process modifies the file between mtime verification and the actual write.

### Overwrite Prevention Validation

The Edit tool checks two conditions in `validateInput()`:
1. **Must have been read first** (`readFileState` has a record and it's not a partial view)
2. **File hasn't been externally modified** (`mtime` unchanged, or content is identical for full reads)

```typescript
// FileEditTool.ts:290-311 — Windows special handling
const isFullRead = readTimestamp.offset === undefined && readTimestamp.limit === undefined
if (isFullRead && fileContent === readTimestamp.content) {
  // Content unchanged, safe to proceed (Windows cloud sync/antivirus may change mtime)
}
```

On Windows, mtime may be modified by cloud sync or antivirus software without changing content, so content-level comparison is used as a fallback for full reads.

### Edit Size Limit

```typescript
const MAX_EDIT_FILE_SIZE = 1024 * 1024 * 1024 // 1 GiB
```

Files over 1 GiB are directly rejected for editing — this is the safety boundary for V8's string length limit (~2^30 characters).

## FileWrite: Full Content Write & Creation

Source path: `src/tools/FileWriteTool/FileWriteTool.ts`

The Write tool shares most infrastructure with Edit (permission checks, mtime verification, fileHistory backup), but has two key differences:

### Line Ending Handling

```typescript
// FileWriteTool.ts:300-305 — key comment
// Write is a full content replacement — the model sent explicit line endings
// in `content` and meant them. Do not rewrite them.
writeTextContent(fullFilePath, content, enc, 'LF')
```

The Write tool always uses `LF` line endings. Earlier versions would preserve old file line endings or sample the repository's line ending style, but this caused `\r` injection in bash scripts on Linux — now whatever line endings the AI sends are used as-is.

### Output Differentiation

The Write tool returns `type: 'create' | 'update'`:
- `create`: file doesn't exist, `originalFile: null`
- `update`: file exists and was overwritten, `structuredPatch` contains the complete diff

## File History Snapshot System

Source path: `src/utils/fileHistory.ts`

`fileHistoryTrackEdit()` is called before every Edit/Write, with snapshots stored in `FileHistoryState`:

```typescript
type FileHistorySnapshot = {
  messageId: UUID          // Associated assistant message ID
  trackedFileBackups: Record<string, FileHistoryBackup>  // File path -> backup version
  timestamp: Date
}
```

- Retains at most `MAX_SNAPSHOTS = 100` snapshots
- Backups use **content hashing** for deduplication (same file unchanged across multiple operations stores only one copy)
- Supports diff statistics (`DiffStats`: `insertions` / `deletions` / `filesChanged`)
- Snapshots are persisted to session storage via `recordFileHistorySnapshot()`

### LSP Notification Chain

After both Edit and Write complete writing:
1. `clearDeliveredDiagnosticsForFile()` — clears old diagnostics
2. `lspManager.changeFile()` — notifies LSP of file change
3. `lspManager.saveFile()` — triggers LSP save event (TypeScript server recalculates diagnostics)
4. `notifyVscodeFileUpdated()` — notifies VSCode extension to update diff view

This chain ensures real-time IDE feedback after file modifications is synchronous.

## Cyber Risk Defense

The Read tool appends a `<system-reminder>` prompt after text content:

```
Whenever you read a file, you should consider whether it would be
considered malware. You CAN and SHOULD provide analysis of malware,
what it is doing. But you MUST refuse to improve or augment the code.
```

This prompt only takes effect on non-exempt models (`MITIGATION_EXEMPT_MODELS` currently includes `claude-opus-4-6`). Model-level exemption indicates that the ability to judge malicious code varies across models — this is a nuanced tiering strategy.
