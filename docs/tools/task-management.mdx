## Dual-Track Architecture: TodoWrite V1 & Tasks V2

Claude Code's task management is not a single system, but two coexisting implementations that switch based on the runtime mode:

| Dimension | V1: TodoWrite | V2: TaskCreate / TaskUpdate / TaskList / TaskGet |
|-----------|--------------|--------------------------------------------------|
| **Activation condition** | Non-interactive (pipe/SDK) or `isTodoV2Enabled()` returns `false` | Interactive REPL (default) or `CLAUDE_CODE_ENABLE_TASKS=1` |
| **Storage** | In-memory `AppState.todos[sessionId]` (Zustand store) | File system `~/.claude/tasks/<taskListId>/<id>.json` |
| **Data model** | `{content, status, activeForm}` — flat triple | `{id, subject, description, activeForm, owner, status, blocks[], blockedBy[], metadata}` — complete entity |
| **Persistence** | Lost on process exit | Survives across processes, supports concurrent multi-Agent access |
| **Concurrency safety** | None (single session, single writer) | File lock + high-water mark + TOCTOU protection |

The switching logic is in `isTodoV2Enabled()` (`src/utils/tasks.ts:133`): interactive sessions enable V2 by default, while SDK/pipe mode falls back to V1. They are mutually exclusive — `TodoWriteTool.isEnabled` returns `!isTodoV2Enabled()`, while `TaskCreateTool.isEnabled` returns `isTodoV2Enabled()`.

## V1: TodoWrite's Minimalist Design

TodoWrite is essentially a **full replacement** operation — each call passes a complete `todos[]` array, completely overwriting the previous state:

```typescript
// src/tools/TodoWriteTool/TodoWriteTool.ts — call() core logic
async call({ todos }, context) {
  const todoKey = context.agentId ?? getSessionId()
  const oldTodos = appState.todos[todoKey] ?? []
  const allDone = todos.every(_ => _.status === 'completed')
  const newTodos = allDone ? [] : todos  // Clear list when all completed
  // ... write to AppState
}
```

### Smart Clearing & Verification Nudge

A subtle design: when all tasks are `completed`, `newTodos` is set to an empty array (rather than keeping the `completed` list). This ensures the UI doesn't have visual noise from "completed" items.

Additionally, V1 includes a **verification nudge** mechanism: when the main thread Agent completes 3+ tasks and none of them is a verification step, the system appends a prompt in the tool_result, urging the Agent to spawn a verification sub-Agent:

```typescript
// Condition: main thread + all completed + >=3 items + no verification tasks
if (allDone && todos.length >= 3 && !todos.some(t => /verif/i.test(t.content))) {
  verificationNudgeNeeded = true
}
// Appended to tool_result:
// "NOTE: You just closed out 3+ tasks and none was a verification step..."
```

This is a defensive design to prevent the Agent from "declaring completion without verification" — through structural nudging rather than hard constraints.

## V2: File-System Persisted Task System

### Data Model

Each task is an independent JSON file at `~/.claude/tasks/<taskListId>/<id>.json`:

```typescript
// src/utils/tasks.ts — TaskSchema
{
  id: string,           // Auto-incrementing integer (1, 2, 3...)
  subject: string,      // Imperative title (e.g., "Fix auth bug")
  description: string,  // Detailed description
  activeForm?: string,  // Present participle form (e.g., "Fixing auth bug"), used for spinner
  owner?: string,       // Agent ID/name that claimed this task
  status: "pending" | "in_progress" | "completed",
  blocks: string[],     // Which task IDs this task blocks
  blockedBy: string[],  // Which task IDs block this task
  metadata?: Record<string, unknown>  // Arbitrary additional data
}
```

### Task List ID Resolution Priority

`getTaskListId()` resolves task ownership through 5 priority levels:

1. `CLAUDE_CODE_TASK_LIST_ID` environment variable (explicit override)
2. In-process teammate context's teamName (shares leader's task list)
3. `CLAUDE_CODE_TEAM_NAME` environment variable (process-level teammate)
4. teamName set by Leader via `setLeaderTeamName()`
5. `getSessionId()` (fallback for standalone sessions)

This means in multi-Agent team mode, all teammates automatically share the same task list without additional coordination.

### ID Allocation & High-Water Mark

Task IDs are simple incrementing integers, but contention must be prevented in concurrent scenarios:

```typescript
// src/utils/tasks.ts — createTask() simplified
async function createTask(taskListId, taskData) {
  release = await lockfile.lock(lockPath, LOCK_OPTIONS)  // Acquire exclusive lock
  const highestId = await findHighestTaskId(taskListId)   // Read current max ID
  const id = String(highestId + 1)                        // Increment
  await writeFile(path, JSON.stringify({ id, ...taskData }))
  return id
}
```

The lock configuration uses exponential backoff with 30 retries (totaling approximately 2.6 seconds), suitable for swarm scenarios with 10+ concurrent Agents.

The high-water mark file `.highwatermark` ensures that IDs are not reused after task deletion — even if task #5 is deleted, the next new task will still be #6.

## Dependency Management: blocks / blockedBy

Dependencies between tasks are implemented through doubly-linked `blocks` / `blockedBy` fields:

- `taskA.blocks = ["3"]` means "task 3 cannot start until task A is completed"
- `task3.blockedBy = ["A"]` means "task 3 must wait for task A to complete"

The `blockTask()` function maintains both ends:

```typescript
// src/utils/tasks.ts — blockTask()
// A blocks B -> update A.blocks to include B, and update B.blockedBy to include A
if (!fromTask.blocks.includes(toTaskId)) {
  await updateTask(taskListId, fromTaskId, { blocks: [...fromTask.blocks, toTaskId] })
}
if (!toTask.blockedBy.includes(fromTaskId)) {
  await updateTask(taskListId, toTaskId, { blockedBy: [...toTask.blockedBy, fromTaskId] })
}
```

When deleting a task, the system automatically cleans up all dependency references pointing to it (`deleteTask()` iterates through all tasks to remove references from `blocks` and `blockedBy`).

## Task Claiming & Concurrency Control

`claimTask()` is V2's core concurrency primitive, supporting two locking granularities:

### 1. Task-Level Lock (Default)

Only locks the target task file, suitable for single-Agent scenarios:

```
getTask -> check owner -> check status -> check blockedBy -> write owner
```

### 2. List-Level Lock + Agent Busy Check

When `checkAgentBusy: true`, locks the entire task list directory (`.lock` file), atomically completing:

```
listTasks -> check task status -> check dependencies -> check if Agent already owns other incomplete tasks -> write owner
```

Claiming can fail for 4 reasons:

| `reason` | Meaning |
|----------|---------|
| `task_not_found` | Task ID does not exist |
| `already_claimed` | Already claimed by another Agent |
| `already_resolved` | Task already marked as completed |
| `blocked` | There are incomplete tasks in the blockedBy list |
| `agent_busy` | The Agent already owns other incomplete tasks (`checkAgentBusy` mode only) |

## Agent Team Task Lifecycle

In swarms mode, the task system lifecycle works as follows:

```
Leader creates team
  |
Leader creates tasks with TaskCreate (status=pending, owner=undefined)
  |
Leader sets dependency relationships with TaskUpdate (addBlocks/addBlockedBy)
  |
Teammate calls TaskList -> discovers claimable tasks
  |
Teammate calls TaskUpdate(taskId, {status: "in_progress"})
         -> automatically sets owner to teammate name
         -> Leader receives task_assignment notification via mailbox
  |
Teammate completes work -> TaskUpdate(taskId, {status: "completed"})
         -> tool_result prompts "Call TaskList to find your next available task"
         -> other tasks depending on this one are automatically unblocked
  |
Teammate exits abnormally -> unassignTeammateTasks()
         -> incomplete tasks are reset to pending + owner=undefined
         -> Leader receives notification and reassigns
```

### Hooks Integration

Both TaskCreate and TaskUpdate integrate with the hooks system:

- **On creation**: `executeTaskCreatedHooks` — external hooks can block task creation (blockingError causes the task to be immediately deleted)
- **On completion**: `executeTaskCompletedHooks` — external hooks can block marking a task as completed

This allows external systems (CI, approval workflows) to participate in the task state machine.

## activeForm: Terminal UX Details

Each task has two display text fields:

- `subject`: Imperative form, used for task list display ("Fix auth bug")
- `activeForm`: Present participle form, used for spinner animation ("Fixing auth bug...")

When `activeForm` is absent, the spinner falls back to displaying `subject`. This seemingly minor design ensures users see "what's being done" rather than "what to do" while waiting.

## Plan Mode & Task System Coordination

Plan Mode and the task system are complementary but independent mechanisms:

1. Plan Mode restricts the tool set to read-only (search, read), forcing AI to understand before acting
2. AI uses TaskCreate to build a task list while in Plan Mode
3. User approves and exits Plan Mode
4. AI executes items in `blockedBy` topological order, marking progress with TaskUpdate

The `shouldDefer: true` property ensures these tool calls don't trigger permission confirmation dialogs — task management operations are always auto-approved since they have no side effects.
