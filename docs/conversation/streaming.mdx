## Why Streaming Is Needed

Imagine AI needs 30 seconds to generate a complete response — if you wait 30 seconds and then display everything at once, the user experience is catastrophic.

Streaming responses let users **see AI's thinking process in real time**:
- Text appears character by character, letting users judge early whether the direction is correct
- Tool call parameters can be previewed as they are generated
- Long tasks don't make users feel like things are "frozen"

## `BetaRawMessageStreamEvent` Core Event Types

The streaming API returns a series of `BetaRawMessageStreamEvent`, where each event type corresponds to a different phase of the streaming response (`src/services/api/claude.ts`):

```
message_start           ← Message start, contains model, initial usage values
  ├── content_block_start   ← Content block start (text / tool_use / thinking)
  │   ├── content_block_delta  ← Incremental data (text_delta / input_json_delta / thinking_delta)
  │   ├── content_block_delta  ← ... continues arriving
  │   └── content_block_stop   ← Content block end, yield AssistantMessage
  ├── content_block_start   ← Next content block...
  │   └── ...
  └── message_delta       ← stop_reason + final usage
message_stop            ← Message end
```

### Event Processing State Machine

`src/services/api/claude.ts:1980-2298` implements a `switch(part.type)` state machine:

| Event Type | Processing Logic | State Change |
|----------|----------|----------|
| `message_start` | Initialize `partialMessage`, record TTFT (time to first byte) | `usage` initialized |
| `content_block_start` | Create content block of corresponding type by `part.index` | `contentBlocks[index]` initialized |
| `content_block_delta` | Incrementally append data by subtype | text / thinking / input accumulated |
| `content_block_stop` | Build complete `AssistantMessage` and yield | Message pushed into `newMessages` |
| `message_delta` | Update stop_reason and final usage | Write back to last message |
| `message_stop` | No-op (stream end marker) | — |

### Content Block Types and Their Incremental Data

The `content_block.type` in `content_block_start` determines how subsequent deltas are processed:

| Content Block Type | Delta Type | Accumulation Logic |
|-----------|-----------|----------|
| `text` | `text_delta` | `text += delta.text` |
| `thinking` | `thinking_delta` + `signature_delta` | `thinking += delta.thinking`, `signature = delta.signature` |
| `tool_use` | `input_json_delta` | `input += delta.partial_json` (incremental JSON string concatenation) |
| `server_tool_use` | `input_json_delta` | Same as tool_use |
| `connector_text` | `connector_text_delta` | Special connector text (controlled by feature flag) |

Key design: all text fields are initialized to empty strings at `content_block_start`, and only accumulated through `content_block_delta`. This is because the SDK sometimes sends duplicate text in both start and delta events.

## Interleaving of Text Chunks and tool_use Blocks

A single AI response may contain multiple content blocks, appearing alternately:

```
content_block_start (text, index=0)     "Let me help you fix this bug."
content_block_delta  (text_delta)       "First..."
content_block_stop  (index=0)
content_block_start (tool_use, index=1) { name: "Read", input: "..." }
content_block_delta  (input_json_delta) '{"file_p' → 'ath":' → '"src/foo.ts"}'
content_block_stop  (index=1)
content_block_start (text, index=2)     "I can see the issue now..."
content_block_stop  (index=2)
```

Each `content_block_stop` triggers one `yield`, pushing a complete AssistantMessage to the consumer. This means a single AI response produces **multiple** `AssistantMessages` — text messages and tool call messages yielded alternately.

The `stop_reason` is only determined at `message_delta` (could be `end_turn`, `tool_use`, `max_tokens`, etc.), so the last message's `stop_reason` is **written back**:

```typescript
// claude.ts:2246 — Direct property modification, not object replacement
// Because the transcript write queue holds a reference to message.message
const lastMsg = newMessages.at(-1)
if (lastMsg) {
  lastMsg.message.usage = usage
  lastMsg.message.stop_reason = stopReason
}
```

## Error Handling in Streaming

### Network Disconnection

Streaming connections rely on SSE (Server-Sent Events). When a connection is interrupted:

1. **Stream idle watchdog**: Periodically checks event intervals, triggers alerts and retries when threshold (stall) is exceeded
2. **Stream abort**: If the watchdog detects prolonged absence of events, throws an error entering the retry flow
3. **Non-streaming fallback**: As a last resort, falls back to non-streaming request (full response in one shot)

```typescript
// claude.ts:2338-2355 — Detecting empty streams
// 1. No events at all → proxy returned a non-SSE response
// 2. Has message_start but no content_block_stop → stream was truncated
```

### API Rate Limiting

When the API returns rate limit errors, the system uses a `withRetry` wrapper for exponential backoff retry. Retry logic considers:
- Error type (429 rate limit vs 500 server error)
- Maximum retry count
- Backoff interval

### Token Exceeded

Two token exceeded scenarios have different handling:

| Scenario | stop_reason | Handling |
|------|------------|----------|
| **Output exceeded** | `max_tokens` | Generate error message, suggest setting `CLAUDE_CODE_MAX_OUTPUT_TOKENS` |
| **Context window exceeded** | `model_context_window_exceeded` | Trigger compaction to compress conversation history, then retry |

```typescript
// claude.ts:2267-2293
if (stopReason === 'max_tokens') {
  yield createAssistantAPIErrorMessage({ error: 'max_output_tokens', ... })
}
if (stopReason === 'model_context_window_exceeded') {
  // Reuse the max_output_tokens recovery path
  yield createAssistantAPIErrorMessage({ error: 'max_output_tokens', ... })
}
```

### Streaming Stall Detection

The system continuously monitors event arrival intervals, detecting "stalls":

```typescript
// claude.ts:1940-1966
const STALL_THRESHOLD_MS = 10_000  // 10 seconds without events is considered a stall
if (timeSinceLastEvent > STALL_THRESHOLD_MS) {
  stallCount++
  totalStallTime += timeSinceLastEvent
  logEvent('tengu_streaming_stall', { stall_duration_ms, stall_count, ... })
}
```

After multiple stalls accumulate, the watchdog may decide to interrupt the stream and trigger a retry.

## Streaming Feedback for Tool Execution

BashTool's command execution is also streaming — pushing output line by line via `onProgress` callbacks:

```
BashTool.call() → runShellCommand() → AsyncGenerator
  ├── Poll output file every second → onProgress(lastLines, allLines, ...)
  ├── yield { type: 'progress', output, fullOutput, elapsedTimeSeconds }
  └── return { code, stdout, interrupted, ... }
```

The UI layer displays command output in real time via the `useToolCallProgress` hook, rather than waiting for the command to fully complete. Long-running commands also support auto-backgrounding (`shouldAutoBackground`).

## Multi-Provider Adaptation

| Provider | Streaming Protocol | Special Handling |
|----------|----------|----------|
| **Anthropic Direct** | Native SSE | Lowest latency, fastest TTFT |
| **AWS Bedrock** | AWS SDK streaming interface | Requires additional beta headers and authentication |
| **Google Vertex** | gRPC → event stream | Adapted via `getMergedBetas()` |
| **Azure** | Anthropic-compatible API | Custom base URL |

All Providers are abstracted behind a unified `Stream<BetaRawMessageStreamEvent>` layer that hides differences. Upper-layer code (QueryEngine, REPL) doesn't need to know which Provider is being used underneath.

### Provider Selection

`src/utils/model/providers.ts`'s `getAPIProvider()` determines which Provider to use based on configuration:

```typescript
// Select based on api_provider configuration:
// "anthropic" → direct connection
// "bedrock"   → AWS SDK
// "vertex"    → Google SDK
// Third-party base URL → auto-detect
```

Details each Provider needs to adapt include: authentication method, beta headers, request parameter format, and error code mapping — but these differences are unified within `claude.ts`'s `queryStream()` function.
