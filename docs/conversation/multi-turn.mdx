## Single-Turn vs Multi-Turn: Architectural Differences

- **Single-turn** (one Agentic Loop): A complete execution of the `query()` function — assemble context → call API → process tool calls → loop until end
- **Multi-turn** (one Session): A session managed by the `QueryEngine` class — spanning dozens of `submitMessage()` calls, lasting hours

`QueryEngine` (`src/QueryEngine.ts:186`) is the **session orchestrator** on top of the single-turn Agentic Loop, managing state far beyond just the message list:

```
QueryEngine Internal State
├── mutableMessages: Message[]         ← Complete conversation history, accumulated across turns
├── readFileState: FileStateCache      ← Read file content cache, avoids redundant reads
├── totalUsage: NonNullableUsage       ← Cumulative token consumption (input/output/cache)
├── permissionDenials: SDKPermissionDenial[]  ← Permission denial records
├── discoveredSkillNames: Set<string>  ← Skills discovered in the current turn
└── abortController: AbortController   ← Session-level abort control
```

## QueryEngine's Core Method: submitMessage()

Each time a user inputs a message, the REPL or SDK calls `submitMessage()`, which executes the complete turn initialization pipeline:

```typescript
// src/QueryEngine.ts:211 — Simplified submitMessage flow
async *submitMessage(prompt, options?): AsyncGenerator<SDKMessage> {
  // 1. Clear turn-level tracking state
  this.discoveredSkillNames.clear()
  
  // 2. Resolve model (user may have switched models mid-session)
  const mainLoopModel = userSpecifiedModel
    ? parseUserSpecifiedModel(userSpecifiedModel)
    : getMainLoopModel()
  
  // 3. Dynamically assemble System Prompt (rebuilt every turn)
  const { defaultSystemPrompt, userContext, systemContext } =
    await fetchSystemPromptParts({ tools, mainLoopModel, mcpClients })
  
  // 4. Wrap permission checks (track each denial)
  const wrappedCanUseTool = async (tool, input, ...) => {
    const result = await canUseTool(tool, input, ...)
    if (result.behavior !== 'allow') {
      this.permissionDenials.push({ tool_name: tool.name, ... })
    }
    return result
  }
  
  // 5. Call core query() function to execute the agentic loop
  yield* query({
    systemPrompt, messages: this.mutableMessages,
    tools, model: mainLoopModel, ...
  })
}
```

Key design: `submitMessage()` is an `async *Generator` — it incrementally yields `SDKMessage`s, letting the caller (REPL/SDK) display progress in real time rather than waiting for the entire turn to complete.

## Session Persistence: JSONL Transcript

Every conversation event is appended to a transcript file (`src/utils/sessionStorage.ts`):

### Storage Path

```
~/.claude/projects/<project-hash>/<session-id>.jsonl
```

- `project-hash` is generated by `getProjectDir(originalCwd)`, grouping sessions from the same project directory into the same subdirectory
- Each record is a single line of JSON (JSONL format), supporting append writes without requiring read-modify-write of the entire file
- Read limit is 50MB (`MAX_TRANSCRIPT_READ_BYTES`), preventing oversized sessions from causing OOM

### Transcript Writer

`TranscriptWriter` (`src/utils/sessionStorage.ts:1200+`) is a write queue ensuring concurrent message appends don't overwrite each other:

```
Write flow:
  appendEntryToFile(sessionId, entry)
    ↓
  ensureCurrentSessionFile()   ← Lazy initialization: file created on first write
    ↓
  Serialize to JSON + newline
    ↓
  appendFile(path, line)       ← Atomic append
    ↓
  If remote persistence is configured:
    persistToRemote(sessionId, entry)
      ├── CCR v2: internalEventWriter('transcript', entry)
      └── v1 Ingress: sessionIngress.appendSessionLog(...)
```

### Session Recovery Pipeline

The recovery flow triggered by the `--resume` parameter (`src/main.tsx:3620+`):

```
1. Parse resume parameter:
   ├── UUID format → getTranscriptPathForSession(uuid)
   ├── .jsonl file path → Use directly
   └── boolean → Most recent session picker
   
2. loadTranscriptFromFile(path)
   ├── Parse by JSONL lines
   ├── Filter for message-type records
   └── Rebuild Message[] array

3. Restore context state:
   ├── restoreCostStateForSession(sessionId)  ← Restore cumulative cost
   ├── Restore agentSetting (user-selected Agent type)
   └── If --rewind-files, restore files to snapshot at specified message

4. Create QueryEngine({ initialMessages: restoredMessages })
   └── Continue conversation from restored messages
```

## Cost Tracking: From API Usage to Dollars

Cost tracking spans three modules, forming a complete record → accumulate → display pipeline:

### Recording Layer: Usage in API Responses

Each `message_delta` event carries a `usage` field (`input_tokens`, `output_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`). `accumulateUsage()` adds incremental usage to the session total.

### Accumulation Layer: cost-tracker.ts

```typescript
// src/cost-tracker.ts — StoredCostState data model
type StoredCostState = {
  totalCostUSD: number                       // Cumulative USD spend
  totalAPIDuration: number                   // Total API call duration (including retries)
  totalAPIDurationWithoutRetries: number     // Pure inference time without retries
  totalToolDuration: number                  // Total tool execution duration
  totalLinesAdded: number                    // Lines of code added
  totalLinesRemoved: number                  // Lines of code removed
  modelUsage: { [modelName: string]: ModelUsage }  // Usage breakdown by model
}
```

`addToTotalSessionCost()` calculates the cost of each API call based on model pricing, accumulating to `totalCostUSD`. Per-model `ModelUsage` supports separate tracking after switching models within the same session.

### Persistence: Retained Across Restarts

```typescript
// Save to project configuration at end of each session
saveCurrentSessionCosts(sessionId)
  → projectConfig.lastCost = totalCostUSD
  → projectConfig.lastSessionId = sessionId
  → projectConfig.lastModelUsage = modelUsage
```

### Budget Circuit Breaker

`QueryEngineConfig.maxBudgetUsd` provides a hard session-level budget cap. In the REPL, when cumulative cost exceeds $5 (`src/screens/REPL.tsx:2208`), a cost reminder dialog appears — this is not a hard block, but a "soft reminder."

## Hot Model Switching

Switching models during a session doesn't lose conversation history — because `mutableMessages` is decoupled from model selection:

```
/model sonnet → setMainLoopModelOverride('claude-sonnet-4-20250514')
  ↓
At the start of the next submitMessage():
  ↓
parseUserSpecifiedModel(userSpecifiedModel)
  → Returns new model configuration
  ↓
fetchSystemPromptParts({ mainLoopModel: newModel })
  → System Prompt reassembled based on new model capabilities
  ↓
query({ model: newModel, messages: this.mutableMessages })
  → Continue conversation with full history + new model
```

When switching models, `contextWindowTokens` and `maxOutputTokens` are also recalculated based on the new model's specifications — for example, switching from Sonnet to Opus may change the context window from 200K to 1M.

## File Snapshots and Rollback

`fileHistoryMakeSnapshot()` (`src/utils/fileHistory.ts`) automatically saves current file contents before each AI modification. Snapshots are bound to a specific `message.id`, enabling `--rewind-files <user-message-id>` to precisely restore files to their state at any point in the conversation — this is finer-grained than git (git only tracks committed content).
