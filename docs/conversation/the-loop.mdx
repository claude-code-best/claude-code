## What Is the Agentic Loop

Traditional chatbot: you ask one question, it gives one answer.  
Claude Code is different: you state a requirement, and it may execute a dozen operations before giving you the final result.

The mechanism behind this is called the **Agentic Loop**, with the core implementation in `src/query.ts`'s `queryLoop()` async generator function (line 241). It is a `while(true)` infinite loop, where each iteration represents one "think → act → observe" cycle.

## Complete Loop Structure

Each iteration of `queryLoop()` (`src/query.ts:307` `while(true)`) includes the following phases:

### Phase 1: Context Pre-Processing Pipeline

Before calling the API, 5 compaction/optimization steps are executed sequentially:

```
messagesForQuery (original messages)
  ↓ applyToolResultBudget()    — Tool result budget truncation (by maxResultSizeChars)
  ↓ snipCompactIfNeeded()      — History Snip compaction (HISTORY_SNIP feature)
  ↓ microcompact()             — Micro-compaction (tool result summarization)
  ↓ applyCollapsesIfNeeded()   — Context collapse (CONTEXT_COLLAPSE feature)
  ↓ autocompact()              — Auto-compaction (triggered when threshold exceeded)
messagesForQuery (processed messages) → Sent to API
```

Each step's output is the input for the next, forming a serial pipeline. Tokens freed by Snip and Microcompact are passed to autocompact's threshold calculation (`snipTokensFreed`), avoiding redundant compaction.

### Phase 2: Streaming API Call (Streaming Loop)

`deps.callModel()` initiates a streaming request (line 659), returning an AsyncGenerator. During streaming:

- **AssistantMessages** are collected into the `assistantMessages[]` array
- **tool_use blocks** are extracted into `toolUseBlocks[]`, setting `needsFollowUp = true`
- **StreamingToolExecutor** begins parallel tool execution during streaming (without waiting for the stream to end)
- Recoverable errors (prompt-too-long, max-output-tokens) are **withheld**, with recovery attempted first

Key guards in streaming callbacks:
- `backfillObservableInput()` (line 763) — backfills observable fields for tool_use blocks (such as file path expansion), but only clones the message when new fields are added, avoiding breaking prompt cache byte consistency
- Streaming fallback detection — if `streamingFallbackOccured`, collected messages are marked as tombstones (line 717), cleared, then retried

### Phase 3: Tool Execution

If `needsFollowUp` is true, the loop doesn't terminate but instead executes tools:

```typescript
// Two tool executors (mutually exclusive)
const toolUpdates = streamingToolExecutor
  ? streamingToolExecutor.getRemainingResults()  // Streaming: get completed + pending
  : runTools(toolUseBlocks, assistantMessages, canUseTool, toolUseContext)
```

Tool results are normalized via `normalizeMessagesForAPI()`, merged with original messages, and enter the **next loop iteration**.

### Phase 4: Terminate or Continue

At the end of each iteration, conditions determine whether to `return` (terminate) or `continue`:

## 4 Continuation Conditions (Recovery Paths)

The loop is not simply "continue if there's tool_use" — it also includes multiple recovery/retry paths:

### 1. Normal Tool Loop
`needsFollowUp = true` → execute tools → new messages appended to `messagesForQuery` → `continue`

### 2. max_output_tokens Recovery (Lines 1191-1255)
When AI output is truncated (`apiError === 'max_output_tokens'`):
- **First time**: Try raising `maxOutputTokens` from default to `ESCALATED_MAX_TOKENS` (64K), no meta message, silent retry
- **Subsequent**: Inject recovery message "Output token limit hit. Resume directly...", retry up to `MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3` times
- After recovery exhaustion, withheld error message is released

### 3. Prompt-Too-Long Recovery (Lines 1088-1186)
When a 413 error is encountered, there are two recovery stages:
- **Context Collapse Drain** (line 1097): Submit all staged collapses, free space and retry. Skipped if the previous round was already a collapse_drain_retry
- **Reactive Compact** (line 1123): Trigger immediate compaction, generate summary and retry. `hasAttemptedReactiveCompact` prevents infinite loops

### 4. Stop Hook Blocking Retry (Lines 1285-1308)
Stop hooks can inject blocking error messages, forcing AI to rethink. New messages (containing blocking errors) are appended to the conversation, `stopHookActive = true`, entering the next iteration.

## Model Fallback

When the primary model is unavailable (`FallbackTriggeredError`, line 897):

1. Collected `assistantMessages` are cleared, tool_use blocks receive synthetic tool_result: "Model fallback triggered"
2. Thinking signature blocks are removed (`stripSignatureBlocks`) — because thinking signatures are model-bound, cross-model replay would cause 400 errors
3. Switch to `fallbackModel`, update `toolUseContext.options.mainLoopModel`
4. Generate system message: "Switched to {fallback} due to high demand for {original}"
5. Re-initiate streaming request

## State Machine: State Object

Each iteration's state is passed via the `State` type (line 204):

```typescript
type State = {
  messages: Message[]                        // Current conversation messages
  toolUseContext: ToolUseContext              // Tool context (including permissions)
  autoCompactTracking: AutoCompactTrackingState  // Compaction tracking
  maxOutputTokensRecoveryCount: number       // Output truncation recovery count
  hasAttemptedReactiveCompact: boolean        // Whether reactive compaction has been attempted
  maxOutputTokensOverride: number | undefined // Output token limit override
  pendingToolUseSummary: Promise<...> | undefined  // Async tool summary
  stopHookActive: boolean | undefined        // Whether Stop hook is active
  turnCount: number                          // Turn count
  transition: Continue | undefined           // Reason for the last continuation
}
```

Each `continue` creates a new State object (immutable update), rather than modifying in place. The `transition` field records why continuation occurred — allowing subsequent iterations to detect specific recovery paths (such as `collapse_drain_retry`) to avoid loops.

## Token Budget (Experimental)

When the `TOKEN_BUDGET` feature is enabled (line 1311), the loop checks token consumption before termination:

- **continuation**: Budget not reached but threshold exceeded → inject nudge message, prompting AI to wrap up faster
- **diminishing_returns**: Diminishing returns detected → early termination
- Budget data comes from `createBudgetTracker()`, accumulated across iterations

## Why Not "Plan Once, Execute in Batch"

<Note>
Source code reveals why Claude Code chose iterative looping:
</Note>

- **Each step produces real information**: `runTools()` returns `toolResults` that the API cannot possibly predict — command output, file contents, error messages
- **Dynamic context management**: Compaction needs are re-evaluated before each iteration (autocompact → microcompact → snip), based on the latest token counts
- **Instant error recovery**: Tool failures don't require starting over — stop hooks can inject blocking errors to let AI correct its strategy
- **User controllable**: `abortController.signal` is checked at multiple checkpoints in the loop (lines 1018, 1048, 1488), allowing users to gracefully interrupt by pressing ESC
- **Cost control**: Token Budget checks at each termination point prevent AI from looping ineffectively

## A Complete Iteration Example

> User: "Find all unused import statements in the project and remove them"

```
Iteration 1: Think → Act
  Pre-processing: No compaction needed (context is short)
  API call: Returns tool_use(Glob, "**/*.ts")
  Tool execution: Returns 42 file paths
  → needsFollowUp = true, continue

Iteration 2: Think → Act
  Pre-processing: 42 file results still within budget
  API call: Returns tool_use(Grep, "import.*from")
  Tool execution: Found 120 imports across 15 files
  → needsFollowUp = true, continue

Iteration 3: Think → Act (multi-turn)
  Pre-processing: 120 Grep results trigger microcompact → summarized
  API call: Returns 3 tool_use(FileEdit, ...)
  Tool execution: Removed 5 unused imports
  → needsFollowUp = true, continue

Iteration 4: Summary
  API call: Returns plain text "Cleaned up 5 unused imports across 3 files"
  → needsFollowUp = false
  → Stop hooks pass
  → return { reason: 'completed' }
```
