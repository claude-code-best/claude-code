## Problem Scenario

You say "refactor this module," and the AI immediately starts modifying code — but you haven't figured out how it plans to change things. By the time you notice the approach is wrong halfway through, it's already too late.

## Plan Mode's Solution

Plan Mode adds a "read-only phase" to the conversation, implemented through two tools that form a closed loop:

<Steps>
  <Step title="EnterPlanMode — Enter Plan Mode">
    AI autonomously determines (or user triggers) that the task needs planning, and calls `EnterPlanModeTool` (`src/tools/EnterPlanModeTool/EnterPlanModeTool.ts:36`). This tool requires **user approval** (`checkPermissions` returns `ask`).
  </Step>
  <Step title="Exploration Phase — Read-Only Tool Set">
    The permission mode switches to `'plan'`, and AI can only use tools where `isReadOnly()` is true (Read, Grep, Glob, Agent, etc.). Write operations are automatically rejected.
  </Step>
  <Step title="ExitPlanMode — Submit Plan for Approval">
    After AI completes exploration, it calls `ExitPlanModeV2Tool` (`src/tools/ExitPlanModeTool/ExitPlanModeV2Tool.ts:147`), submitting the plan file for user review. This is the second **user approval** checkpoint.
  </Step>
  <Step title="Resume Execution — Full Tool Permissions">
    After user approval, the permission mode reverts to its pre-entry state, and AI executes according to the plan.
  </Step>
</Steps>

## Automatic Permission Narrowing & Restoration

### Entry: `prepareContextForPlanMode()`

Core logic of `EnterPlanModeTool.call()` (line 77):

```typescript
// 1. Record transition state (save previous mode)
handlePlanModeTransition(currentMode, 'plan')

// 2. Switch permission context to plan mode
context.setAppState(prev => ({
  ...prev,
  toolPermissionContext: applyPermissionUpdate(
    prepareContextForPlanMode(prev.toolPermissionContext),
    { type: 'setMode', mode: 'plan', destination: 'session' },
  ),
}))
```

What `prepareContextForPlanMode()` (`src/utils/permissions/permissionSetup.ts`) does:
- Creates a new `ToolPermissionContext` with `mode` set to `'plan'`
- In plan mode, the tool's `isReadOnly()` check becomes the sole admission criterion
- If the user's default mode is `'auto'`, it also activates the classifier's side effects

### Exit: Permission Restoration + Prompt-Based Permissions

`ExitPlanModeV2Tool`'s exit logic does two critical things:

**1. Restore Permission Mode**

Restores to the pre-entry mode via `handlePlanModeTransition()` and `applyPermissionUpdate()`.

**2. Inject Prompt-Based Permissions**

This is Plan Mode's most elegant design — AI can declare in its plan the categories of commands it needs to execute:

```typescript
// ExitPlanModeV2Tool's inputSchema
allowedPrompts: z.array(z.object({
  tool: z.enum(['Bash']),
  prompt: z.string().describe('Semantic description, e.g. "run tests"'),
})).optional()
```

When AI submits its plan, if it declares `allowedPrompts: [{ tool: 'Bash', prompt: 'run tests' }]`, after user approval, "run tests" type Bash commands are auto-approved — no more individual confirmations needed.

## Plan File Persistence

Plan content is written to a disk file (path determined by `getPlanFilePath()`), which is fundamentally different from "AI says something and then starts executing":

1. `ExitPlanModeV2Tool`'s `normalizeToolInput` reads plan content from disk, injecting it into `input.plan` and `input.planFilePath`
2. The plan file is **user-editable** — users can modify AI's proposal before approval
3. The `planWasEdited` field marks whether the user modified the plan, affecting subsequent tool_result echo
4. `persistFileSnapshotIfRemote()` saves file snapshots in remote scenarios

## Plan Approval in Teammate Scenarios

In Agent Swarms (`isAgentSwarmsEnabled()`) mode, plan approval has an additional collaboration workflow:

```typescript
// If this is a Teammate role
if (isTeammate()) {
  // Send plan to Team Leader's mailbox for approval
  const requestId = generateRequestId()
  writeToMailbox(getTeamName(), {
    type: 'plan_approval_request',
    plan, requestId, ...
  })
  // Return awaitingLeaderApproval: true
  // Team Leader approves via mailbox notification to Teammate
}
```

This means in swarm mode, the plan may not be approved by the direct user but by the Team Leader.

## When to Use Plan Mode

`EnterPlanModeTool`'s Prompt (`src/tools/EnterPlanModeTool/prompt.ts`) defines two sets of trigger criteria — the external version is more aggressive (encourages planning), while the internal version is more restrained (only used when truly ambiguous):

| Scenario | External Version | Internal Version |
|----------|-----------------|------------------|
| Fix a typo | Skip | Skip |
| Add a delete button | **Enter** (involves multiple files) | **Skip** (path is clear) |
| Refactor auth system | **Enter** | **Enter** (high-impact refactoring) |
| "Start doing X" | — | **Skip** (start directly) |
| Architecture decision (Redis vs in-memory cache) | **Enter** | **Enter** (truly ambiguous) |

## Plan Mode + Task System

Plan Mode is typically used in conjunction with the task system:

1. While in plan mode, AI creates implementation steps as a task list (`TodoWrite`)
2. User approves the plan (including the task list)
3. After exiting plan mode, AI executes items from the task list sequentially
4. Users can track progress through the task list

## Complete Lifecycle

```
User: "Refactor this module"
  |
AI determines planning is needed -> calls EnterPlanModeTool
  | User approval (Ask dialog)
handlePlanModeTransition(default, 'plan')  // Save default
prepareContextForPlanMode()                // Create read-only context
  |
AI uses Read/Grep/Glob/Agent to explore the codebase
  | (potentially 10+ rounds of read-only tool calls)
AI forms a proposal -> calls ExitPlanModeV2Tool({
  allowedPrompts: [
    { tool: 'Bash', prompt: 'run tests' },
    { tool: 'Bash', prompt: 'install dependencies' }
  ]
})
  | User approves plan (can edit plan file)
Restore permission mode -> inject prompt-based permissions
  |
AI uses full tool set to execute plan, "run tests" and similar commands are auto-approved
```
