## The Second Line of Defense Beyond Permissions

The permission system decides "whether this command can execute"; the sandbox decides "what it can do during execution."

Even if a command passes permission approval, the sandbox can still restrict its behavior. Together they form two layers of defense in depth:
- **Permission layer** (application level): Checks before tool calls, decides whether to prompt for approval
- **Sandbox layer** (OS level): Enforces constraints at the process level; even if AI generates a malicious command, it cannot break through

## Execution Chain: From User Input to Sandbox Wrapping

The complete execution path of a Bash command:

```
User input -> BashTool.call()
           -> shouldUseSandbox(input) --- Needs sandbox?
           -> Shell.exec(command, { shouldUseSandbox })
           -> SandboxManager.wrapWithSandbox(command)
           -> spawn(wrapped_command)  --- Actual process creation
```

The key determination happens in `shouldUseSandbox()` (`src/tools/BashTool/shouldUseSandbox.ts`), which performs the following checks:

1. **Global switch**: `SandboxManager.isSandboxingEnabled()` — checks platform support + dependency completeness + user settings
2. **Explicit skip**: If `dangerouslyDisableSandbox: true` and policy allows (`allowUnsandboxedCommands`), skip sandbox
3. **Exclusion list**: Users can configure `sandbox.excludedCommands` in `settings.json`; matching commands skip sandbox
4. **Default behavior**: If none of the above conditions are met, **enter sandbox**

## `shouldUseSandbox()` Decision Logic Details

```typescript
// src/tools/BashTool/shouldUseSandbox.ts
function shouldUseSandbox(input: Partial<SandboxInput>): boolean {
  // 1. Globally disabled -> skip directly
  if (!SandboxManager.isSandboxingEnabled()) return false

  // 2. Explicitly disabled + policy allows -> skip
  if (input.dangerouslyDisableSandbox && 
      SandboxManager.areUnsandboxedCommandsAllowed()) return false

  // 3. No command -> skip
  if (!input.command) return false

  // 4. Matches exclusion list -> skip
  if (containsExcludedCommand(input.command)) return false

  // 5. All other cases -> must be sandboxed
  return true
}
```

The `containsExcludedCommand()` matching mechanism is worth noting — it doesn't just do simple prefix matching but supports three modes:

| Mode | Example | Match Behavior |
|------|---------|----------------|
| **Exact match** | `npm run lint` | Full equality |
| **Prefix match** | `npm run test:*` | Prefix + space or full equality |
| **Wildcard** | `docker*` | Uses `matchWildcardPattern` |

For compound commands (e.g., `docker ps && curl evil.com`), the system first splits into subcommands and checks each one. It also iteratively strips environment variable prefixes (`FOO=bar bazel ...`) and wrapper commands (`timeout 30 bazel ...`) until reaching a fixed point — preventing bypass through nested wrapping.

## Sandbox Configuration Model

Sandbox configuration comes from the `sandbox` field in `settings.json` (`src/entrypoints/sandboxTypes.ts`):

```jsonc
{
  "sandbox": {
    "enabled": true,                     // Master switch
    "autoAllowBashIfSandboxed": true,     // Auto-approve commands in sandbox (skip approval)
    "allowUnsandboxedCommands": true,     // Allow dangerouslyDisableSandbox
    "failIfUnavailable": false,           // Error and exit if sandbox dependencies are missing
    
    "network": {
      "allowedDomains": ["github.com"],   // Network allowlist
      "deniedDomains": [],                // Network blocklist
      "allowLocalBinding": true,          // Allow localhost binding
      "httpProxyPort": 8888               // HTTP proxy port (MITM)
    },
    
    "filesystem": {
      "allowWrite": ["~/projects"],       // Additional writable paths
      "denyWrite": ["~/.ssh"],            // Write-denied paths
      "denyRead": [],                     // Read-denied paths
      "allowRead": []                     // Re-allow within denyRead
    },
    
    "excludedCommands": ["docker", "npm:*"]  // Commands that skip sandbox
  }
}
```

`SandboxSettingsSchema` defines the complete Zod validation rules, including some undocumented settings such as `enabledPlatforms` (restricts sandbox to specific platforms only).

## Platform Implementation Differences

### macOS: sandbox-exec (Seatbelt)

macOS uses Apple's Seatbelt sandbox (`sandbox-exec` command), which is macOS's native process isolation mechanism.

Execution flow:
1. `SandboxManager.wrapWithSandbox()` calls `BaseSandboxManager` from `@anthropic-ai/sandbox-runtime`
2. The runtime generates a Seatbelt profile (based on network/filesystem rules from configuration)
3. Wraps the original command via `sandbox-exec -p <profile> -- <command>`
4. Seatbelt enforces constraints at the kernel level

Network isolation implementation:
- Intercepts HTTP/HTTPS requests through a proxy port
- Domain allowlist/blocklist filtering at the proxy layer
- Unix sockets can be configured with separate allowed paths

### Linux: bubblewrap (bwrap) + seccomp

Linux uses `bubblewrap` (bwrap) to create namespace isolation, combined with seccomp for system call filtering:

Dependencies (`apt install`):
| Package | Purpose |
|---------|---------|
| `bubblewrap` | Creates mount/PID/network namespaces |
| `socat` | Network proxy (HTTP/SOCKS) |
| `libseccomp` / seccomp filter | Filters Unix socket system calls |

bwrap implementation differences:
- **Does not support glob path patterns** (macOS Seatbelt does) — permission rules with globs on Linux trigger warnings
- Leaves 0-byte mount-point files (e.g., `.bashrc`) in the current directory after execution, requiring `cleanupAfterCommand()` cleanup
- seccomp cannot filter Unix sockets by path (only allow-all or deny-all), differing from macOS per-path allowlisting

### Platform Support Matrix

| Feature | macOS | Linux | WSL |
|---------|-------|-------|-----|
| Sandbox engine | sandbox-exec (Seatbelt) | bubblewrap + seccomp | WSL2 only |
| File glob | Full support | Only `/**` suffix | Same as Linux |
| Network Unix socket by path | Yes | No | No |
| Dependency check | ripgrep | bwrap + socat + ripgrep + seccomp | Same as Linux |

## Sandbox Initialization Flow

```
REPL/SDK startup
  -> main.tsx -> init.ts
  -> SandboxManager.initialize(sandboxAskCallback)
    -> detectWorktreeMainRepoPath()     // Detect git worktree, allow main repo .git
    -> convertToSandboxRuntimeConfig()  // Build SandboxRuntimeConfig
    -> BaseSandboxManager.initialize()  // Start underlying runtime
    -> settingsChangeDetector.subscribe() // Subscribe to settings changes, dynamically update config
```

`convertToSandboxRuntimeConfig()` (`src/utils/sandbox/sandbox-adapter.ts`) handles the conversion from user settings to runtime configuration:

1. **Network rules**: Extracts domains from `WebFetch(domain:...)` permission rules -> `allowedDomains`
2. **Filesystem rules**: Extracts paths from `Edit(...)` / `Read(...)` permission rules -> `allowWrite` / `denyWrite` / `denyRead`
3. **Security hardening**:
   - Automatically adds project directory to `allowWrite`
   - Automatically adds `settings.json` path to `denyWrite` (prevents sandbox escape)
   - Automatically adds `.claude/skills` to `denyWrite` (prevents skill injection)
   - Detects bare git repo attack vectors, protecting `HEAD`/`objects`/`refs`

## `dangerouslyDisableSandbox` Design Tradeoffs

The naming of this parameter itself conveys the design intent — it's not "disable sandbox" but "**dangerously disable sandbox**."

Dual insurance mechanism:
1. **Caller side**: The model can set `dangerouslyDisableSandbox: true` in BashTool's `inputSchema`
2. **Policy side**: Administrators can completely disable this parameter via `allowUnsandboxedCommands: false` (enterprise deployment scenario)

```typescript
// Even if AI requests dangerouslyDisableSandbox, the policy layer can still override
if (input.dangerouslyDisableSandbox && 
    SandboxManager.areUnsandboxedCommandsAllowed()) {
  return false  // Only truly skip sandbox when policy allows
}
```

`autoAllowBashIfSandboxed` further complements this model: when enabled, **commands within the sandbox automatically receive execution permission** without per-command approval. This is based on a trust assumption — if the OS-level sandbox already restricts the command's capabilities, application-level per-command approval becomes redundant.

## Sandbox Violation Handling

When a command attempts to violate sandbox constraints:

1. The runtime captures violation events (file/network access denied)
2. `SandboxManager.annotateStderrWithSandboxFailures()` injects `<sandbox_violations>` tags in the output
3. The UI layer cleans up the display via `removeSandboxViolationTags()`
4. Violation events are persisted via `SandboxViolationStore` for auditing

## Complete Execution Chain Example

Using `npm install` as an example:

```
1. User types in REPL -> Claude decides to call BashTool
2. BashTool.validateInput() -> passes
3. BashTool.checkPermissions() -> check permission rules
   |-- autoAllowBashIfSandboxed = true and sandbox available -> auto-approve
   +-- Otherwise -> prompt user for confirmation
4. BashTool.call() -> runShellCommand()
5. shouldUseSandbox({ command: "npm install" })
   |-- SandboxManager.isSandboxingEnabled() -> true
   |-- dangerouslyDisableSandbox -> undefined
   +-- containsExcludedCommand() -> false (unless user configured npm exclusion)
   -> Result: true, needs sandbox
6. Shell.exec() -> SandboxManager.wrapWithSandbox("npm install")
   |-- macOS: sandbox-exec -p <generated-profile> -- bash -c 'npm install'
   +-- Linux: bwrap ... bash -c 'npm install'
7. spawn(wrapped_command) -> child process executes within sandbox
8. Execution complete -> SandboxManager.cleanupAfterCommand()
   |-- Clean up bwrap residual files (Linux)
   +-- scrubBareGitRepoFiles() (security cleanup)
9. Result returned to Claude -> displayed to user
```
