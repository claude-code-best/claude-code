## Three Permission Behaviors

For every tool call, the system makes one of three decisions:

| Behavior | Meaning | Return Type | Typical Scenario |
|----------|---------|-------------|------------------|
| **Allow** | Auto-approved, user unaware | `{ behavior: 'allow', updatedInput, decisionReason }` | Read accessing project files |
| **Ask** | Displays confirmation dialog | `{ behavior: 'ask', message, suggestions, metadata }` | Bash executing unknown command |
| **Deny** | Directly rejected | `{ behavior: 'deny', message, decisionReason }` | Attempting a forbidden command |

These behaviors are defined by the `PermissionResult` type (`src/utils/permissions/PermissionResult.ts`).

## Five-Layer Rule Sources

Rules converge from 5 sources (`PERMISSION_RULE_SOURCES`, `permissions.ts:109`), from highest to lowest priority:

```
1. session        — User manually authorized in current conversation ("Always allow")
2. cliArg         — Command-line --allow/--deny arguments
3. command        — Skill tool's allowedTools allowlist
4. projectSettings — .claude/settings.json (team-shared)
5. userSettings   — ~/.claude/settings.json (cross-project)
6. policySettings — Enterprise admin-deployed policies (user cannot override)
```

Each source maintains three arrays: `alwaysAllowRules[source]`, `alwaysAskRules[source]`, `alwaysDenyRules[source]`.

The rule data structure is `PermissionRule`:
```typescript
{
  source: PermissionRuleSource      // Which layer it comes from
  ruleBehavior: 'allow' | 'ask' | 'deny'
  ruleValue: {
    toolName: string                // e.g., "Bash", "mcp__server1"
    ruleContent?: string            // e.g., "git *", "src/**"
  }
}
```

## Rule Matching Engine

### Three-Dimensional Matching

`permissions.ts` implements three matching dimensions:

**1. Tool Name Matching** (`toolMatchesRule()`, line 238)

Matches the entire tool, only when the rule has no `ruleContent`:
```typescript
// Exact match
rule "Bash" -> matches BashTool
rule "mcp__server1" -> matches all tools from that MCP Server (server level)
rule "mcp__server1__*" -> wildcard match (same as above)
```

MCP tools use `getToolNameForPermissionCheck()` to get the matching name, supporting both prefixed (`mcp__server__tool`) and non-prefixed modes.

**2. Command Pattern Matching** (BashTool's `checkPermissions()`)

BashTool parses command patterns via `preparePermissionMatcher()` (`Tool.ts:514`):
```json
{"tool": "Bash", "ruleContent": "git *"}  -> matches "git commit -m 'fix'"
```

Commands are parsed via AST (`readOnlyValidation.ts` uses tree-sitter bash), extracting the first subcommand for matching.

**3. Path Matching** (file tool `checkPermissions()`)

Read/Edit/Write tools extract file paths via `getPath()`, matching against glob patterns in `ruleContent`:
```json
{"tool": "Edit", "ruleContent": "src/**"}  -> matches "src/utils/foo.ts"
```

### Complete Permission Check Flow

The permission check for each tool call (`canUseTool()` -> `checkPermissions()`) goes through these steps:

```
1a. Blanket deny check
    getDenyRuleForTool() -> tool name fully matches deny rule?
    | hit -> deny (tool filtered out at getTools() stage)

1b. Blanket allow check
    toolAlwaysAllowedRule() -> tool name fully matches allow rule?
    | hit -> allow

2. Tool's own checkPermissions()
    Each tool has custom logic:
    - BashTool: readOnlyValidation -> sandbox check -> AST parsing -> pattern matching
    - FileEditTool: path allowlist check
    - SkillTool: safe properties allowlist + exact/prefix matching
    | returns PermissionResult

3. Hook system
    executePermissionRequestHooks() -> PreToolUse hook can override
    | hook returns deny -> deny
    | hook returns ask -> escalate to ask

4. Ask rule check
    getAskRules() -> hit -> ask

5. Default behavior
    Based on current permissionMode
    - 'default': most tools ask
    - 'plan': write operations deny, read operations allow
    - 'bypass': all allow
```

## Permission Modes

| Mode | `PermissionMode` Value | Use Case | Behavior |
|------|----------------------|----------|----------|
| **Default** | `'default'` | Daily use | Sensitive operations confirmed individually |
| **Plan Mode** | `'plan'` | Exploration phase | Read-only, no writing (`isReadOnly()` check) |
| **Auto** | `'auto'` | Trust AI | Automatic decisions via transcript classifier |
| **Bypass** | `'bypassPermissions'` | Full trust | All operations auto-approved (requires explicit `--dangerously-skip-permissions`) |

Plan Mode switching is triggered by `EnterPlanModeTool.call()`:
```typescript
// EnterPlanModeTool.ts:88
context.setAppState(prev => ({
  ...prev,
  toolPermissionContext: applyPermissionUpdate(
    prepareContextForPlanMode(prev.toolPermissionContext),
    { type: 'setMode', mode: 'plan', destination: 'session' },
  ),
}))
```

On exit, `ExitPlanModeV2Tool` restores the previous mode.

## Denial Tracking: Deadloop Prevention

`src/utils/permissions/denialTracking.ts` implements a denial tracking mechanism:

```typescript
const DENIAL_LIMITS = {
  maxDenialsPerTool: 3,        // Max consecutive denials per tool
  cooldownPeriodMs: 30_000,    // 30-second cooldown period
}
```

When AI is consecutively denied the same type of operation up to the limit:
1. `recordDenial()` records the denial, increments counter
2. `shouldFallbackToPrompting()` detects consecutive denials, returns true
3. System injects a message to AI: "Your previous tool call was rejected..."
4. AI is forced to change strategy, avoiding the deadloop of "repeatedly requesting the same rejected operation"

Calling `recordSuccess()` resets the counter on successful operations.

## Runtime Rule Updates

Permission rules can be dynamically updated at runtime (`applyPermissionUpdate()`, `PermissionUpdate.ts`):

```typescript
type PermissionUpdate =
  | { type: 'addRule', behavior, rule, destination }
  | { type: 'removeRule', behavior, rule, destination }
  | { type: 'setMode', mode, destination }
```

When a user selects "Always allow" in the Ask dialog, the system calls `persistPermissionUpdates()` to write the rule to the corresponding level's settings file (project/user/managed), while simultaneously updating the in-memory `toolPermissionContext`.
