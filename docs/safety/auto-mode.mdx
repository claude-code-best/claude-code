## Overview

Auto mode is a permission mode in Claude Code that puts AI into a **continuous autonomous execution** state. Unlike the traditional mode (where every sensitive operation triggers a permission dialog waiting for user approval), auto mode uses an AI classifier (transcript classifier) to automatically judge whether each tool call is safe, enabling an uninterrupted execution experience.

```
Permission mode hierarchy:

default  ->  auto  ->  bypassPermissions
(per-item confirmation) (AI classifier approval) (all approved)
```

## Core Architecture

### 1. AI Classifier (yoloClassifier)

The classifier is the core of auto mode, located in `src/utils/permissions/yoloClassifier.ts`.

Each tool call is evaluated by the classifier, returning one of three verdicts:

| Verdict | Meaning | Handling |
|---------|---------|----------|
| **allow** | Operation is safe | Execute directly, user unaware |
| **deny** | Operation is dangerous | Reject execution, return denial message to AI |
| **ask** | Cannot determine | Fall back to traditional permission dialog |

The classifier's judgment is based on the complete conversation context (transcript), not just the individual command — it can understand the operation's intent and context.

### 2. Two-Stage Classification Pipeline

The classifier supports two-stage XML classification (`classifyYoloActionXml`):

```
Stage 1 — "fast" (quick judgment)
|-- max_tokens=64, stop_sequences trigger immediate yes/no
|-- If allow -> approve directly (fastest path)
+-- If block -> proceed to Stage 2

Stage 2 — "thinking" (deep reasoning)
|-- chain-of-thought reasoning
|-- Reduces false positives
+-- Final decision: allow / deny / ask
```

Both stages share the same system prompt and user content, leveraging API prompt caching (1-hour TTL) for performance optimization.

Mode can be configured:
- `'both'` (default) — run both stages
- `'fast'` — run Stage 1 only
- `'thinking'` — run Stage 2 only

### 3. Classifier Result Types

```typescript
// src/types/permissions.ts
type YoloClassifierResult = {
  thinking?: string          // Classifier's reasoning process
  shouldBlock: boolean       // Whether to block
  reason: string             // Decision reason
  unavailable?: boolean      // Whether classifier is unavailable
  transcriptTooLong?: boolean // Whether conversation exceeds context window
  model: string              // Classifier model used
  stage?: 'fast' | 'thinking' // Which stage made the decision
  // ... token usage, latency, and other monitoring fields
}
```

## Safety Mechanisms

### Dangerous Permission Stripping

When entering auto mode, the system calls `stripDangerousPermissionsForAutoMode()` (`permissionSetup.ts:510`), removing all allow rules that could bypass the classifier.

Stripped rule types (`dangerousPatterns.ts`):

| Rule Type | Example | Stripping Reason |
|-----------|---------|------------------|
| **Bash code execution** | `Bash(python:*)`, `Bash(node:*)` | Interpreters can execute arbitrary code, bypassing classifier review |
| **Shell entry points** | `Bash(bash:*)`, `Bash(sh:*)` | Direct shell access equals unrestricted access |
| **Agent rules** | `Agent(*)` | Any Agent allow rule bypasses classifier approval for sub-agents |
| **PowerShell code execution** | `PowerShell(node:*)` | Same logic as Bash |
| **Privilege escalation** | `Bash(sudo:*)`, `Bash(eval:*)` | Can execute arbitrary commands |

Stripped rules are temporarily stored in `strippedDangerousRules` and restored via `restoreDangerousPermissions()` when exiting auto mode.

### Model Support Detection

Not all models support auto mode. `modelSupportsAutoMode()` (`src/utils/betas.ts`) checks whether the current model has safe classification capability. Models without support cannot enter auto mode.

### Circuit Breaker Mechanism

`autoModeState.ts` maintains a circuit breaker flag:

```typescript
let autoModeCircuitBroken = false  // Controlled by remote configuration
```

When remote configuration (GrowthBook `tengu_auto_mode_config.enabled`) is set to `'disabled'`, the circuit breaker triggers, preventing auto mode entry and continued use. This provides Anthropic with remote emergency shutdown capability.

## Mode Switching State Management

### Entering Auto Mode

`transitionPermissionMode()` (`permissionSetup.ts:597`) handles all mode switches:

```
1. Check if auto mode gate is enabled (isAutoModeGateEnabled)
2. Set autoModeActive = true
3. Call stripDangerousPermissionsForAutoMode() to strip dangerous rules
4. Inject Auto Mode system prompt into conversation
```

### Exiting Auto Mode

```
1. Set autoModeActive = false
2. Set needsAutoModeExitAttachment = true (triggers exit notification)
3. Call restoreDangerousPermissions() to restore stripped rules
4. Inject "Exited Auto Mode" prompt into conversation
```

### Activation Paths

Auto mode can be activated via:
- CLI argument `--enable-auto-mode`
- `autoMode` configuration in settings.json
- Plan mode defaults to auto mode semantics (`useAutoModeDuringPlan`, default true)
- SDK control messages
- Shift+Tab toggle in REPL

## System Prompts

### On Entry (Full Instructions)

Instructions injected into the conversation (`messages.ts:3464`):

> Auto mode is active. The user chose continuous, autonomous execution. You should:
>
> 1. **Execute immediately** — Implement directly, make reasonable assumptions
> 2. **Minimize interruptions** — Make routine decisions yourself, reduce questions
> 3. **Prefer action over planning** — Default to coding directly, don't enter plan mode
> 4. **Expect course corrections** — User can redirect at any time
> 5. **Do not take overly destructive actions** — Deleting data/modifying production systems still requires confirmation
> 6. **Avoid data exfiltration** — Don't proactively share secrets/internal documents

### During Continuous Operation (Sparse Instructions)

Brief reminder injected in subsequent turns:

> Auto mode still active. Execute autonomously, minimize interruptions, prefer action over planning.

### On Exit (Exit Instructions)

> You have exited auto mode. Ask clarifying questions when the approach is ambiguous rather than making assumptions.

## Collaboration with Plan Mode

Plan mode defaults to using auto mode semantics (`getUseAutoModeDuringPlan()`, default true). This means:

- When entering plan mode, if auto mode is available, the classifier is also activated
- `isAutoModeActive()` is the authoritative signal (`prePlanMode`/`strippedDangerousRules` are unreliable)
- Exiting plan mode also exits auto mode simultaneously

## Classifier Unavailability Fallback Strategy

When the classifier API is unavailable (`unavailable: true` or `transcriptTooLong: true`):

- Does not directly allow — falls back to traditional permission dialog (ask)
- Sends message to AI: "{model} is temporarily unavailable, so auto mode cannot determine the safety of {toolName} right now."
- Deterministic errors (e.g., conversation too long) don't retry, degrade directly

## Classifier Prompt Templates

The classifier's behavior is controlled by three prompt files, located in `src/utils/permissions/yolo-classifier-prompts/`. These files are inlined as string constants via `require()` at build time and are not modifiable at runtime.

### auto_mode_system_prompt.txt

Main system prompt defining the classifier's role, classification process, and decision categories. Contains:

- **Classification process**: Understand operation -> check user intent -> assess risk
- **BLOCK always**: External code execution, irreversible deletion, unauthorized persistence, security weakening, privilege escalation, network services
- **BLOCK unless clear intent**: Writing outside CWD, system package management, git push, large-scale changes
- **ALLOW safe operations**: Reading files, searching, git read-only commands, test/lint/build, editing within CWD
- `<permissions_template>` placeholder — replaced at runtime with specific permission template (external or anthropic)
- Trailing `Use the classify_result tool to report your classification.` — replaced in XML mode with `<block>` tag format instructions

### permissions_external.txt

External user version of the permission template. Three `<user_*_to_replace>` tags wrap default rules (bullet format), and user custom rules **fully replace** defaults:

```
<user_allow_rules_to_replace>
- Default allow rule 1
- Default allow rule 2
</user_allow_rules_to_replace>
```

- **allow**: 9 default rules (read-only commands, version checks, git read-only, test/lint/build, package install, CWD file operations, etc.)
- **soft_deny**: 10 default rules (external code execution, recursive deletion, shell config modifications, privilege escalation, network services, etc.)
- **environment**: 4 environment descriptions (terminal environment, auto mode context, dev tools available, language/framework agnostic)

`getDefaultExternalAutoModeRules()` extracts bullet items from this file, output by the `claude auto-mode defaults` command.

### permissions_anthropic.txt

Anthropic internal version of the permission template. Default rules are **outside** the tags, tags are empty, and user custom rules are **appended**:

```
- Default rules (outside tags, always active)
<user_allow_rules_to_replace>
</user_allow_rules_to_replace>
```

Compared to the external version, additionally includes:
- Cloud CLI read-only commands (aws describe, gcloud describe, kubectl get, etc.)
- Infrastructure-as-code plan commands (terraform plan, pulumi preview, etc.)
- Corresponding deny rules (cloud resource create/modify/delete, IaC apply, production environment access, etc.)

### Template Replacement Flow

```
buildYoloSystemPrompt()
  |-- BASE_PROMPT.replace('<permissions_template>', EXTERNAL/ANTHROPIC_TEMPLATE)
  |-- .replace(<user_allow_rules_to_replace>, userAllow ?? defaults)
  |-- .replace(<user_deny_rules_to_replace>, userDeny ?? defaults)
  +-- .replace(<user_environment_to_replace>, userEnvironment ?? defaults)
```

- External template: user settings **replace** corresponding tag content when non-empty, otherwise retain defaults
- Internal template: user settings are **appended** after defaults (tags are empty at the end)
