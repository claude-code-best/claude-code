## Tool vs Skill: The Essential Difference

| | Tool | Skill |
|---|---|---|
| Granularity | A single atomic operation (read file, execute command) | A complete workflow (code review, create PR) |
| Trigger Method | AI autonomously selects | User `/skill-name` or AI auto-matches via `SkillTool` |
| Essence | TypeScript execution logic | **Declarative encapsulation of Prompt + permission configuration** |
| Registration Location | `src/tools.ts` → `getTools()` | `src/commands.ts` → `getCommands()` |
| Executor | Each Tool's `call()` method | `SkillTool.call()` → two branches (inline / fork) |

The core insight of Skills: **the key to complex tasks is not code logic, but Prompt quality**. A code review Skill doesn't need a review engine — it just needs to tell the AI "what to review, in what order, what output format" — Skills encapsulate this "expertise" as reusable Markdown.

## Five Sources and Loading Pipeline of Skills

### 1. Built-in Commands

Hardcoded in the `COMMANDS` memoized array at `src/commands.ts:258`, containing 70+ commands (`/commit`, `/review`, `/compact`, etc.). These are TypeScript modules rather than Markdown, but implement the same `Command` interface (`src/types/command.ts`).

### 2. Bundled Skills (Compiled at Build Time)

Registered via `registerBundledSkill()` (`src/skills/bundledSkills.ts:53`) during module initialization. Key characteristics:

- **Lazy file extraction**: If a Skill declares `files` (reference files), they are extracted to a temporary directory on first invocation (`getBundledSkillExtractDir()`), using `O_NOFOLLOW | O_EXCL` to prevent symlink attacks (`safeWriteFile`, line 186)
- **Closure-level memoize**: Concurrent calls share the same extraction promise, avoiding race condition writes
- Source marked as `source: 'bundled'`, enjoying **non-truncatable** privilege in Prompt budget

### 3. Disk Skills (`.claude/skills/`)

Loaded by `loadSkillsFromSkillsDir()` (`src/skills/loadSkillsDir.ts:407`), this is the most important loading path:

```
Managed policy: $MANAGED_DIR/.claude/skills/     (policySettings)
User global:    ~/.claude/skills/                 (userSettings)
Project-level:  .claude/skills/                   (projectSettings, traverses up to home)
Additional dirs: paths specified by --add-dir under .claude/skills/
```

**Loading protocol**: Only recognizes the `skill-name/SKILL.md` directory format; single-file `.md` is no longer supported. Loading flow:

1. `readdir` scans directory → keeps only `isDirectory()` or `isSymbolicLink()` entries
2. Looks for `SKILL.md` in each subdirectory, skips if not found
3. `parseFrontmatter()` parses the YAML header, extracting `whenToUse`, `allowedTools`, `context`, and other fields
4. `parseSkillFrontmatterFields()` (line 185) uniformly parses 17 frontmatter fields
5. `createSkillCommand()` (line 270) constructs a `Command` object

**Deduplication mechanism**: Uses `realpath()` to resolve symlinks and obtain canonical paths (`getFileIdentity`, line 118), avoiding duplicate loading through symlinks or overlapping parent directories.

### 4. MCP Skills (Dynamic Discovery)

Builders are registered via `registerMCPSkillBuilders()`, and MCP Server prompts are converted to `Command` objects by `mcpSkillBuilders.ts`. Marked as `loadedFrom: 'mcp'`.

**Security boundary**: MCP Skills' Prompt content is **prohibited from executing inline shell commands** (`loadSkillsDir.ts:374` guard with `loadedFrom !== 'mcp'`), because remote content is untrusted.

### 5. Legacy Commands (`/commands/` Directory)

Backward-compatible old format, loaded by `loadSkillsFromCommandsDir()` (line 566). Supports both the `SKILL.md` directory format and single `.md` file format.

## Complete Frontmatter Fields

A complete `SKILL.md` frontmatter (`parseSkillFrontmatterFields`, line 185):

```yaml
---
name: code-review                    # Display name (overrides directory name)
description: Systematic code review  # Description (or extracted from first Markdown paragraph)
when_to_use: "User says review code, find bugs"  # Basis for AI auto-matching
allowed-tools:                       # Tool whitelist
  - Read
  - Grep
  - Glob
argument-hint: "<file-or-directory>" # Argument hint
arguments: [path]                    # Declarative argument names (for $ARGUMENTS substitution)
model: opus                          # Model override
effort: high                         # Effort level
context: fork                        # Execution mode: inline (default) | fork
agent: code-reviewer                 # Specified Agent definition file
user-invocable: true                 # Whether user can invoke via /
disable-model-invocation: false      # Prohibit AI from invoking autonomously
version: "1.0"                       # Version number
paths:                               # File path patterns for conditional activation
  - "src/**/*.ts"
hooks:                               # Hook configuration
  PreToolUse:
    - command: ["echo", "checking"]
shell: ["bash"]                      # Shell execution environment
---
```

After parsing, 17 fields are extracted, among which `allowedTools`, `model`, and `effort` dynamically modify `toolPermissionContext` at execution time.

## Two Execution Paths: Inline vs Fork

SkillTool (`src/tools/SkillTool/SkillTool.ts:332`) branches in `call()` based on `command.context`:

### Inline Mode (Default)

The Skill's Prompt content is injected as a **UserMessage**, continuing execution in the main conversation flow:

1. `processPromptSlashCommand()` handles argument substitution (`$ARGUMENTS`) and shell command expansion (`` !`...` ``)
2. `${CLAUDE_SKILL_DIR}` is replaced with the absolute path of the Skill's directory
3. `${CLAUDE_SESSION_ID}` is replaced with the current session ID
4. Returns `newMessages` (injected into conversation flow) + `contextModifier` (modifies permission context)

`contextModifier` (line 776) does three things:
- **Tool whitelist injection**: Merges `allowedTools` into `alwaysAllowRules.command`
- **Model switching**: `resolveSkillModelOverride()` handles model override, preserving the `[1m]` suffix to avoid 200K window truncation
- **Effort level override**: Modifies `effortValue`

### Fork Mode (`context: fork`)

The Skill executes in an **independent sub-agent** (`executeForkedSkill`, line 122):

1. `prepareForkedCommandContext()` builds an isolated Agent definition and Prompt
2. `runAgent()` starts a sub-agent loop with an independent token budget
3. Progress of tool usage is reported via `onProgress` callback
4. Results are extracted via `extractResultText()`, and all sub-agent messages are released after extraction (`agentMessages.length = 0`)
5. Finally, state is cleaned up via `clearInvokedSkillsForAgent()`

Fork mode is suitable for scenarios requiring strong isolation (such as long-running review tasks), avoiding pollution of the main conversation's context.

## Permission Model: Safe Properties Whitelist

`checkPermissions()` (line 433) implements a four-tier permission check:

```
1. Deny rule matching (supports exact match and prefix:* wildcards)
   ↓ Not matched
2. Official marketplace Skill auto-pass (plugin + isOfficialMarketplaceName)
   ↓ Not matched
3. Allow rule matching
   ↓ Not matched
4. Safe Properties whitelist check (skillHasOnlySafeProperties, line 911)
   ↓ Has non-safe properties
5. Ask user for confirmation (with both exact match and prefix match suggestion rules)
```

**Safe Properties** (`SAFE_SKILL_PROPERTIES`, line 876) is a whitelist of 28 property names. Any **meaningful property value** not in the whitelist (excluding `undefined`, `null`, empty arrays, empty objects) triggers a permission request. This is a **positive security** design — newly added properties require permission by default in the future.

## Prompt Budget: 1% Context Window Truncation Strategy

When Skill listings are injected into the System Prompt, there is a strict character budget (`prompt.ts`):

- **Budget calculation**: `contextWindowTokens × 4 chars/token × 1%` (approximately 8000 characters)
- **Per-item limit**: `MAX_LISTING_DESC_CHARS = 250` characters (truncated to `...` if exceeded)
- **Bundled Skills are non-truncatable**: They always retain their full description; when budget is insufficient, only non-bundled ones are truncated
- **Degradation strategy**:
  1. Try full descriptions → Over budget?
  2. Bundled retain full, non-bundled split remaining budget equally → Each description below 20 characters?
  3. Non-bundled retain only names

`formatCommandsWithinBudget()` (`prompt.ts:70`) implements this three-tier degradation.

## Dynamic Discovery and Conditional Activation

### File Path-Based Dynamic Discovery

`discoverSkillDirsForPaths()` (`loadSkillsDir.ts:861`) is triggered during file operations:

1. Starting from the operated file path, **traverses upward** to CWD (not including CWD itself)
2. Looks for `.claude/skills/` directories at each level
3. Deduplicates using `realpath`, filters gitignored directories with `git check-ignore`
4. Sorts by path depth (**deeper first**), Skills closer to the file have higher priority

### Conditional Activation (paths frontmatter)

Skills with `paths` patterns are not immediately available at load time; instead they are stored in a `conditionalSkills` Map. When an operated file path matches a Skill's paths pattern (using the `ignore` library for gitignore-style matching), that Skill is **activated** — moved from `conditionalSkills` to `dynamicSkills`.

This means a testing Skill that only activates on `*.test.ts` is completely invisible normally, and only appears when AI reads or edits test files.

## Usage Frequency Ranking

`recordSkillUsage()` (`skillUsageTracking.ts`) uses an exponential decay algorithm to calculate Skill ranking scores:

```
score = usageCount × max(0.5^(daysSinceUse / 7), 0.1)
```

- **7-day half-life**: Usage from a week ago is weighted at half
- **Minimum 0.1 floor**: Prevents old but frequently used Skills from sinking completely
- **60-second debounce**: Multiple invocations of the same Skill within 1 minute count as one, reducing file I/O

Ranking data is persisted in the `skillUsage` field of the global configuration.

## Remote Skill Loading (Experimental)

Controlled by the `EXPERIMENTAL_SKILL_SEARCH` feature flag, supports loading `_canonical_<slug>` format Skills from remote sources (AKI/GCS/S3):

1. `validateInput()` intercepts canonical names via `stripCanonicalPrefix()`
2. `executeRemoteSkill()` (line 970) loads SKILL.md from a remote URL
3. Supports `gs://`, `https://`, `s3://` and other URL protocols
4. Content is injected directly after frontmatter stripping and `${CLAUDE_SKILL_DIR}` replacement
5. Registered to compaction preservation state via `addInvokedSkill()`, ensuring recovery after compaction
6. Remote Skills do not go through `processPromptSlashCommand` — no `!command` substitution, no `$ARGUMENTS` expansion

## Complete Lifecycle Summary

```
Disk SKILL.md
  ↓ parseFrontmatter()
  ↓ parseSkillFrontmatterFields() → 17 fields
  ↓ createSkillCommand() → Command object
  ↓ Deduplication (realpath + seenFileIds)
  ↓ Conditional Skill → conditionalSkills Map (waiting for path match activation)
  ↓ getSkillDirCommands() memoize cache
  ↓ getAllCommands() merge local + MCP
  ↓ formatCommandsWithinBudget() → Truncated Skill listing injected into System Prompt
  ↓ AI selects matching Skill
  ↓ SkillTool.validateInput() → Name validation + existence check
  ↓ SkillTool.checkPermissions() → Four-tier permission check
  ↓ SkillTool.call() → inline or fork execution
  ↓ contextModifier() → Inject allowedTools + model + effort
  ↓ recordSkillUsage() → Update usage frequency ranking
```
