## Execution Engine: execCommandHook

`execCommandHook()` (`src/utils/hooks.ts:829-1417`) is the execution core for command-type Hooks:

```
execCommandHook(hook, hookEvent, hookName, jsonInput, signal)
  ├── Shell selection: hook.shell ?? DEFAULT_HOOK_SHELL
  │   ├── bash: spawn(cmd, [], { shell: gitBashPath | true })
  │   └── powershell: spawn(pwsh, ['-NoProfile', '-NonInteractive', '-Command', cmd])
  ├── Variable substitution
  │   ├── ${CLAUDE_PLUGIN_ROOT} → pluginRoot path
  │   ├── ${CLAUDE_PLUGIN_DATA} → plugin data directory
  │   └── ${user_config.X} → user configuration values
  ├── Environment variable injection
  │   ├── CLAUDE_PROJECT_DIR
  │   ├── CLAUDE_ENV_FILE (SessionStart/Setup/CwdChanged/FileChanged)
  │   └── CLAUDE_PLUGIN_OPTION_* (plugin options)
  ├── stdin write: jsonInput + '\n'
  ├── Timeout: hook.timeout * 1000 ?? 600000ms (10 minutes)
  └── Async detection: Check if stdout first line is {"async":true}
```

### Async Hook Detection Protocol

If the first line of a Hook process's stdout is `{"async":true}`, the system converts it to a background task (`hooks.ts:1199-1246`):

```typescript
const firstLine = firstLineOf(stdout).trim()
if (isAsyncHookJSONOutput(parsed)) {
  executeInBackground({
    processId: `async_hook_${child.pid}`,
    asyncResponse: parsed,
    ...
  })
}
```

Background Hooks are registered to `AsyncHookRegistry` via `registerPendingAsyncHook()`, and upon completion notify the main thread via `enqueuePendingNotification()`.

### asyncRewake: Hook Waking the Model

`asyncRewake` mode Hooks bypass `AsyncHookRegistry`. When the Hook exit code is 2, it injects a message via `enqueuePendingNotification()` in `task-notification` mode, waking an idle model (via `useQueueProcessor`) or injecting a `queued_command` attachment when busy.

## Hook Output JSON Schema

Synchronous Hook output follows a strict Zod schema (`src/types/hooks.ts:49-567`):

```json
{
  "continue": false,                    // Whether to continue execution
  "suppressOutput": true,               // Hide stdout
  "stopReason": "Security check failed",// Reason when continue=false
  "decision": "approve" | "block",      // Global decision
  "reason": "Explanation",              // Decision reason
  "systemMessage": "Warning content",   // System message injected into context
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow" | "deny" | "ask",
    "permissionDecisionReason": "Matched a safety rule",
    "updatedInput": { ... },            // Modified tool input
    "additionalContext": "Extra context" // Injected into conversation
  }
}
```

### hookSpecificOutput Per Event

| Event | Specific Fields | Purpose |
|------|---------|------|
| `PreToolUse` | `permissionDecision`, `updatedInput`, `additionalContext` | Intercept/modify tool input |
| `UserPromptSubmit` | `additionalContext` | Inject additional context |
| `PostToolUse` | `additionalContext`, `updatedMCPToolOutput` | Modify MCP tool output |
| `SessionStart` | `initialUserMessage`, `watchPaths` | Set initial message and file monitoring |
| `PermissionDenied` | `retry` | Indicate whether to retry |
| `Elicitation` | `action`, `content` | Control user input dialog |

## Hook Matching Mechanism: getMatchingHooks

`getMatchingHooks()` (`hooks.ts:1685-1956`) finds matching Hooks from all sources:

### Multi-Source Merging

```
getHooksConfig()
  ├── getHooksConfigFromSnapshot()    ← Hooks in settings.json (user/project/local)
  ├── getRegisteredHooks()            ← SDK-registered callback Hooks
  ├── getSessionHooks()               ← Agent/Skill pre-registered session Hooks
  └── getSessionFunctionHooks()       ← Runtime function Hooks
```

### Matching Rules

The `matcher` field supports three modes (`matchesPattern()`, `hooks.ts:1428-1463`):

```
"Write"              → Exact match
"Write|Edit"         → Pipe-separated multi-value match
"^Bash(git.*)"       → Regex match
"*" or ""            → Wildcard (matches all)
```

### if Condition Filtering

Hooks can specify an `if` condition, triggering only on specific inputs. `prepareIfConditionMatcher()` (`hooks.ts:1472-1503`) pre-compiles matchers:

```json
{
  "hooks": [{
    "command": "check-git-branch.sh",
    "if": "Bash(git push*)"
  }]
}
```

The `if` condition is parsed using `permissionRuleValueFromString`, supporting the same syntax as permission rules (tool name + argument pattern). Bash tools also use tree-sitter for AST-level command parsing.

### Hook Deduplication

The same Hook command may be duplicated across different configuration tiers (user/project/local). The system deduplicates using a `pluginRoot\0command` Map, retaining the **last merged tier**.

## Workspace Trust Check

**All Hooks require workspace trust** (`shouldSkipHookDueToTrust()`, `hooks.ts:286-296`). This is a defense-in-depth measure — preventing a malicious repository's `.claude/settings.json` from executing arbitrary commands without trust.

```typescript
// In interactive mode, all Hooks require trust
const hasTrust = checkHasTrustDialogAccepted()
return !hasTrust
```

In SDK non-interactive mode, trust is implicit (`getIsNonInteractiveSession()` returns true, skipping the check).

## Source Code Mapping of Four Hook Capabilities

### 1. Intercepting Operations (PreToolUse)

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny"
  }
}
```

`processHookJSONOutput()` maps `permissionDecision` to `result.permissionBehavior = 'deny'` and sets `blockingError`, preventing tool execution.

### 2. Modifying Behavior (updatedInput / updatedMCPToolOutput)

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "updatedInput": { "command": "npm test -- --bail" }
  }
}
```

`updatedInput` replaces the original tool input; `updatedMCPToolOutput` (PostToolUse event) replaces MCP tool return values — useful for filtering sensitive data.

### 3. Injecting Context (additionalContext / systemMessage)

- `additionalContext` → Injected as a user message via `createAttachmentMessage({ type: 'hook_additional_context' })`
- `systemMessage` → Injected as a system warning, displayed directly to the user

### 4. Controlling Flow (continue / stopReason)

```json
{ "continue": false, "stopReason": "Build failed, stopping execution" }
```

`continue: false` sets `preventContinuation = true`, preventing the Agent from continuing with subsequent operations.

## Session Hook Lifecycle

Agent and Skill frontmatter Hooks are registered via `registerFrontmatterHooks()` (`runAgent.ts:567-575`), bound to the agent's session ID. When the Agent ends, they are cleaned up via `clearSessionHooks()`.

```typescript
// runAgent.ts:567 — Register agent's frontmatter Hooks
registerFrontmatterHooks(rootSetAppState, agentId, agentDefinition.hooks, ...)

// runAgent.ts:820 — Cleanup in finally block
clearSessionHooks(rootSetAppState, agentId)
```

This ensures Agent A's Hooks don't leak into Agent B's execution.
