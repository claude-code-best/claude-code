## Architecture Overview: From Configuration to Available Tools

```
settings.json: { mcpServers: { "my-db": { command: "npx", args: [...] } } }
  ↓
getAllMcpConfigs()                    ← Merge user/project/local three-level configuration
  ↓
useManageMCPConnections()             ← React Hook managing connection lifecycle
  ↓
connectToServer(name, config)         ← memoize cache (lodash memoize)
  ├── Create Transport (stdio/sse/http/...)
  ├── new Client()                    ← @modelcontextprotocol/sdk
  ├── client.connect(transport)       ← Timeout control (MCP_TIMEOUT, default 30s)
  └── Return MCPServerConnection      ← { connected | failed | needs-auth | pending }
  ↓
fetchToolsForClient(client)           ← LRU(20) cache
  ├── client.request({ method: 'tools/list' })
  └── Each tool wrapped as MCPTool    ← Unified Tool interface
  ↓
assembleToolPool()                    ← Merge built-in tools + MCP tools
  ↓
Tool name format: mcp__<serverName>__<toolName>  ← buildMcpToolName()
```

## 7 Transport Layer Implementations

`connectToServer()` (`client.ts:596-1643`) dispatches to different Transport implementations based on `config.type`:

| Transport Type | Transport Class | Use Case | Authentication |
|----------|-------------|---------|---------|
| `stdio` (default) | `StdioClientTransport` | Local subprocess | None |
| `sse` | `SSEClientTransport` | Remote SSE service | `ClaudeAuthProvider` + OAuth |
| `http` | `StreamableHTTPClientTransport` | HTTP streaming | `ClaudeAuthProvider` + OAuth |
| `sse-ide` | `SSEClientTransport` | IDE integration | lockfile token |
| `ws-ide` | `WebSocketTransport` | IDE WebSocket | `X-Claude-Code-Ide-Authorization` |
| `ws` | `WebSocketTransport` | WebSocket service | session ingress token |
| `claudeai-proxy` | `StreamableHTTPClientTransport` | claude.ai proxy | OAuth bearer + 401 retry |

### stdio Transport Process Management

stdio-type MCP servers run as subprocesses, with cleanup using a **signal escalation strategy** (`client.ts:1431-1564`):

```
SIGINT (100ms) → SIGTERM (400ms) → SIGKILL
```

Total cleanup time is capped at 600ms, preventing MCP server shutdown from blocking CLI exit.

### Remote Transport Authentication State Machine

SSE/HTTP types use `ClaudeAuthProvider` to implement the OAuth authentication flow. When authentication fails, it enters a `needs-auth` state and writes to a 15-minute TTL cache file (`mcp-needs-auth-cache.json`), avoiding repeated authentication prompts.

```
Connection attempt → 401 Unauthorized
  ↓
handleRemoteAuthFailure()
  ├── logEvent('tengu_mcp_server_needs_auth')
  ├── setMcpAuthCacheEntry(name)         ← Write 15min TTL cache
  └── return { type: 'needs-auth' }      ← UI displays authentication prompt
```

## Connection Cache and Reconnection Mechanism

`connectToServer` uses lodash `memoize` to cache connection objects, with cache key `${name}-${JSON.stringify(config)}`.

### Cache Invalidation Triggers

When a connection closes (`client.onclose`), all related caches are cleared (`client.ts:1376-1404`):

```typescript
client.onclose = () => {
  const key = getServerCacheKey(name, serverRef)
  fetchToolsForClient.cache.delete(name)      // Tool cache
  fetchResourcesForClient.cache.delete(name)  // Resource cache
  fetchCommandsForClient.cache.delete(name)   // Command cache
  connectToServer.cache.delete(key)           // Connection cache
}
```

### Connection Degradation Detection

Remote transports have a **consecutive error counter** (`client.ts:1229`):

```typescript
let consecutiveConnectionErrors = 0
const MAX_ERRORS_BEFORE_RECONNECT = 3
```

After 3 consecutive terminal errors (ECONNRESET, ETIMEDOUT, EPIPE, etc.), the transport is proactively closed to trigger reconnection. For HTTP transports, session expiry is also detected (404 + JSON-RPC code -32001).

### Request-Level Timeout Protection

Each HTTP request uses an independent `setTimeout` timeout (`wrapFetchWithTimeout`, `client.ts:493`), rather than a shared `AbortSignal.timeout()`. The reason is that Bun's GC for AbortSignal.timeout is lazy — each request consumes ~2.4KB of native memory, which is not reclaimed until 60s even if the request completes in milliseconds.

```typescript
const controller = new AbortController()
const timer = setTimeout(c => c.abort(...), MCP_REQUEST_TIMEOUT_MS, controller)
timer.unref?.()  // Don't block process exit
```

## Tool Discovery: From MCP to Tool Interface

`fetchToolsForClient()` (`client.ts:1745-2000`) uses `memoizeWithLRU` cache (limit 20), converting MCP tools to Claude Code's unified Tool interface:

```typescript
const fullyQualifiedName = buildMcpToolName(client.name, tool.name)
// Result: "mcp__my-db__query"
```

### Tool Description Truncation

MCP tool descriptions have a 2048 character limit (`MAX_MCP_DESCRIPTION_LENGTH`). MCP servers generated from OpenAPI have been observed with 15-60KB description documents.

### Tool Capability Annotations

Each MCP tool is automatically annotated based on `tool.annotations`:

| Annotation | Maps To | Meaning |
|------|--------|------|
| `readOnlyHint` | `isReadOnly()` + `isConcurrencySafe()` | Read-only, can run in parallel |
| `destructiveHint` | `isDestructive()` | Destructive operation |
| `openWorldHint` | `isOpenWorld()` | Open world (not enumerable) |
| `title` | `userFacingName()` | Display name |

### MCP Tool Permission Checks

MCP tools default to returning `{ behavior: 'passthrough' }` (`client.ts:1816-1834`), meaning they always enter the permission confirmation flow. Tool names use the `mcp__` prefix for exact matching of permission rules.

## MCP Tool Execution Pipeline

```
AI generates tool_use: { name: "mcp__my-db__query", input: { sql: "..." } }
  ↓
MCPTool.call()                               ← client.ts:1835
  ├── ensureConnectedClient()                ← Ensure connection is valid (reconnect)
  ├── callMCPToolWithUrlElicitationRetry()   ← With Elicitation retry
  │   ├── client.request({ method: 'tools/call' })
  │   ├── Handle image results (resize + persist)
  │   └── Content truncation (mcpContentNeedsTruncation)
  ├── McpSessionExpiredError → Retry once
  └── Return { data: content, mcpMeta }
```

### Session Expiry Auto-Retry

HTTP transport MCP sessions may expire. When `McpSessionExpiredError` is detected, an automatic retry occurs once (`client.ts:1862`), since `ensureConnectedClient()` has already cleared the cache and established a new connection.

### Content Truncation and Persistence

Large MCP tool outputs are truncated via `truncateMcpContentIfNeeded`, and binary content (images) is written to files via `persistBinaryContent`, returning file paths. Images are automatically resized (`maybeResizeAndDownsampleImageBuffer`).

## MCP Connection Concurrency Control

```typescript
// Local server concurrent connections
getMcpServerConnectionBatchSize()    // Default 3

// Remote server concurrent connections
getRemoteMcpServerConnectionBatchSize()  // Default 20
```

Local MCP servers (stdio) are heavyweight subprocesses, defaulting to 3 concurrent connections. Remote servers are lightweight HTTP requests, allowing 20 concurrent connections.

## Practical Configuration Example

```json
// MCP configuration in settings.json
{
  "mcpServers": {
    "my-database": {
      "command": "npx",
      "args": ["@my-org/db-mcp-server"],
      "env": { "DB_URL": "postgres://..." }
    },
    "remote-api": {
      "type": "http",
      "url": "https://api.example.com/mcp"
    }
  }
}
```

After configuration, `mcp__my-database__query` and `mcp__remote-api__*` tools will appear in AI's tool list — using the same permission check pipeline and UI rendering as built-in tools.
