## One-Line Definition

Claude Code is an **agentic coding system that runs in your local terminal**. It's not a suggestion-giving chatbot — it directly reads code, modifies files, runs commands, and debugs programs in your project directory, with full shell capabilities.

## Technical Positioning: Terminal-Native Agentic System

Understanding Claude Code comes down to three key terms:

| Positioning Keyword | Meaning |
|---------------------|---------|
| **Terminal-native** | Native CLI application, not an IDE plugin, not a web interface, not an API wrapper |
| **Agentic** | AI autonomously decides tool call chains, not a "one-question-one-answer" chat mode |
| **Coding system** | Oriented toward the full software engineering workflow, not a general-purpose Q&A tool |

**Architectural-level** differences from similar tools (not a feature checklist):

| Tool | Architecture Pattern | Runtime Location | Tool Execution |
|------|---------------------|------------------|----------------|
| **Claude Code** | Terminal-native agentic loop | Local process | Direct shell execution |
| Cursor / Copilot | IDE-integrated autocomplete + chat | Within IDE process | LSP / IDE API |
| Aider | CLI chat -> git patch | Local process | Primarily file operations |
| ChatGPT / Claude.ai | Cloud chat + artifacts | Browser/cloud | Sandbox container |

Core difference: Claude Code has **full shell access** — meaning it can do anything you can do in the terminal, but also needs corresponding security mechanisms to constrain this capability.

## End-to-End Example: From Input to Output

When you type `bun run dev has a TypeScript error, help me fix it` in the terminal, what happens in the system?

```
+-------------------------------------------------------------+
| 1. Entry Layer (cli.tsx -> main.tsx)                          |
|    feature() = false, MACRO injection, start Commander.js CLI |
+-------------------------------------------------------------+
| 2. Interaction Layer (REPL.tsx — React/Ink)                   |
|    PromptInput captures user input -> UserMessage joins       |
|    conversation                                               |
+-------------------------------------------------------------+
| 3. Orchestration Layer (QueryEngine.ts)                       |
|    Manages turn lifecycle, token budget, compaction triggers  |
+-------------------------------------------------------------+
| 4. Core Loop (query.ts — Agentic Loop)                        |
|    Assemble context -> call API -> receive streaming response |
|    -> parse tool calls -> permission check -> execute tools   |
|    -> return results -> call API again -> loop                |
+-------------------------------------------------------------+
| 5. Tool Execution (BashTool.call / FileEditTool.call / ...)   |
|    Actual execution: read files, run commands, search code... |
+-------------------------------------------------------------+
| 6. Communication Layer (claude.ts -> Anthropic API)           |
|    Streaming HTTP, supports Bedrock/Vertex/Azure multi-       |
|    provider                                                   |
+-------------------------------------------------------------+
```

For this specific error-fixing scenario, a typical agentic loop might contain multiple rounds of tool calls:

| Turn | AI Decision | Tool Call | Result |
|------|------------|-----------|--------|
| 1 | First check the error message | `Bash("bun run dev 2>&1 | head -30")` | TypeScript error output |
| 2 | Locate the file | `Read("src/utils/foo.ts")` | Source code content |
| 3 | Search related type definitions | `Grep("interface Foo", "src/")` | Type definition location |
| 4 | Fix the code | `FileEdit(old, new)` | Code modified |
| 5 | Verify the fix | `Bash("bun run dev 2>&1 | head -10")` | Compilation passed |

Every step is autonomously decided by the AI — it decides which tool to use, what parameters to pass, and when to stop. This is what "agentic" means.

## What It Is NOT

- **Not an IDE plugin**: No graphical interface, no dependency on VS Code or any IDE
- **Not an API wrapper**: It has its own tool system, permission model, context engineering, and session management
- **Not a chatbot**: Output is not plain text but actual file modifications and command executions
- **Not a mindless executor**: Every sensitive operation has permission checks and user confirmation steps

## Entry Point Dissection

The actual code entry point is `src/entrypoints/cli.tsx`, which does three key things:

```typescript
// 1. Inject runtime polyfill — feature() always returns false
const feature = (_name: string) => false;

// 2. Inject build-time macros
globalThis.MACRO = { VERSION: "2.1.888", BUILD_TIME: ..., };

// 3. Declare build target
globalThis.BUILD_TARGET = "external";  // External build (not Anthropic internal)
globalThis.BUILD_ENV = "production";
globalThis.INTERFACE_TYPE = "stdio";   // Standard I/O interaction
```

Then control flow passes to `src/main.tsx`:
1. Commander.js parses command-line arguments
2. Initializes authentication, telemetry, and policy restrictions
3. Loads tool list (`getTools()`)
4. Starts REPL (`launchRepl()`) or pipe mode (`-p`)

## Why the Terminal

The terminal is not a limitation but a choice. It brings unique capabilities:

- **Full shell access**: Can run any command-line tool without writing a plugin for each capability
- **Project-native**: Works directly in the project directory, understands file system structure and git status
- **Composability**: Pipe mode (`echo "..." | claude -p`) allows embedding in CI/CD and automation workflows
- **Low latency**: No Electron overhead, React/Ink rendered TUI responds extremely fast

The tradeoff is users need to be comfortable with the command-line interface — but precisely because of this, it attracts developers who need **real control over their development environment**.
