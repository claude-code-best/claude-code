## Five-Layer Architecture

Claude Code is divided top-to-bottom into five layers, each with clear responsibilities and well-defined boundaries:

<Frame caption="Claude Code Five-Layer Architecture">
  <img src="/docs/images/architecture-layers.png" alt="Claude Code Five-Layer Architecture Diagram" />
</Frame>

| Layer | Responsibility | Entry Source Code | Keywords |
|-------|---------------|-------------------|----------|
| **Interaction Layer** | Terminal UI, user input, message display | `src/screens/REPL.tsx` | React/Ink, PromptInput |
| **Orchestration Layer** | Multi-turn conversation, session persistence, cost tracking | `src/QueryEngine.ts` | QueryEngine, transcript |
| **Core Loop Layer** | Single turn: send request -> get response -> execute tools -> loop | `src/query.ts` | Agentic Loop, State |
| **Tool Layer** | AI's "hands" — read/write files, execute commands | `src/tools.ts` -> `src/Tool.ts` | Tool interface, MCP |
| **Communication Layer** | Streaming communication with Claude API | `src/services/api/claude.ts` | Streaming, Provider |

## Source Code Trace of a Core Data Flow

<Frame caption="Core Data Flow">
  <img src="/docs/images/data-flow.png" alt="Claude Code Core Data Flow" />
</Frame>

The entire system's operation can be condensed into one core data flow. Here is the source code path for each step:

### 1. User Input -> REPL

`src/screens/REPL.tsx` is a terminal UI component built with React/Ink. User input is processed by `processUserInput()` (`src/utils/processUserInput/processUserInput.ts`), supporting slash commands, file attachments, images, and more.

### 2. QueryEngine Orchestration

`src/QueryEngine.ts` is the middle layer between REPL and `query()`, managing:
- **Session state**: Message array, tool permission context (`ToolPermissionContext`), file history snapshots
- **Cost tracking**: `accumulateUsage()` / `getTotalCost()` accumulates token usage
- **Transcript persistence**: `recordTranscript()` serializes the conversation to disk, supporting `--resume`
- **File history**: `fileHistoryMakeSnapshot()` creates snapshots before modifications, supporting undo

Key method: `queryEngine.query()` constructs `QueryParams` and calls the `query()` async generator.

### 3. Agentic Loop (`src/query.ts`)

`query()` is an `AsyncGenerator` where each iteration of the `while(true)` loop contains:

```
1. Context preprocessing pipeline:
   applyToolResultBudget -> snipCompact -> microcompact -> contextCollapse -> autocompact

2. Streaming API call:
   deps.callModel() -> AsyncGenerator<StreamEvent | Message>
   Collects assistantMessages[], toolUseBlocks[]

3. Tool execution:
   StreamingToolExecutor (parallel) or runTools (serial)
   -> toolResults[]

4. Terminate/continue decision:
   needsFollowUp ? continue : return { reason }
```

The complete state machine passes `State` type (`src/query.ts:204`) between iterations, containing 10 fields (messages, autoCompactTracking, maxOutputTokensRecoveryCount, etc.).

### 4. Tool Layer (`src/tools.ts` -> `src/Tool.ts`)

`getAllBaseTools()` (`src/tools.ts:191`) assembles the 50+ tool list, which passes through `filterToolsByDenyRules()` permission filtering before being sent to the API.

Each tool implements the `Tool<Input, Output, Progress>` interface (`src/Tool.ts:362`), with the core method chain:
```
validateInput() -> canUseTool() (UI layer) -> checkPermissions() -> call() -> ToolResult
```

### 5. Communication Layer (`src/services/api/claude.ts`)

The API client supports 4 Providers:
- **Anthropic Direct**: Default
- **AWS Bedrock**: `ANTHROPIC_BEDROCK_BASE_URL`
- **Google Vertex**: `ANTHROPIC_VERTEX_PROJECT_ID`
- **Azure**: Via custom base URL

`deps.callModel()` initiates streaming requests, returning `BetaRawMessageStreamEvent` event streams. Supports Prompt Cache (`cache_control`), thinking blocks, and multi-turn tool use.

## Four Core Design Principles

<AccordionGroup>
  <Accordion title="Streaming-First">
    All API communication is streaming — `deps.callModel()` returns an AsyncGenerator, and users see AI "typing out" responses character by character. StreamingToolExecutor starts executing tools in parallel during streaming, without waiting for the stream to end. During model fallback, already-collected assistantMessages are marked as tombstone and cleared, retrying the entire streaming request.
  </Accordion>
  <Accordion title="Tool as Capability">
    Each tool is a `Tool<Input, Output, Progress>` structural type, created via the `buildTool()` factory. `getTools()` assembles tools on every API call (not globally cached), because `isEnabled()` may change with runtime state. MCP tools are tagged with their source via the `mcpInfo` field and support server-level blanket deny.
  </Accordion>
  <Accordion title="Permission as Boundary">
    Every tool call goes through dual `validateInput() -> checkPermissions()` checks. Permission rules converge from 5 sources (session -> project -> user -> managed -> default), supporting matching by tool name, command pattern, path prefix, and more. Plan Mode switches to read-only mode via `prepareContextForPlanMode()`, automatically restoring on exit.
  </Accordion>
  <Accordion title="Context as Memory">
    The System Prompt is dynamically assembled by `fetchSystemPromptParts()`, including CLAUDE.md, git status, date, MCP server list. Auto-compact evaluates token thresholds before each iteration, triggering compression when exceeded. Compressed summaries replace original messages via `buildPostCompactMessages()`, with `taskBudgetRemaining` accumulating across compaction boundaries.
  </Accordion>
</AccordionGroup>
