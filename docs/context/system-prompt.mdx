## From Array to API Call: The Complete System Prompt Pipeline

The System Prompt in Claude Code is not a hardcoded piece of text, but a **`string[]` array** (branded type `SystemPrompt`, defined at `src/utils/systemPromptType.ts:8`), which is assembled, chunked, cache-tagged, and then sent to the API.

### Three-Stage Pipeline

```
getSystemPrompt()          →  string[]       (Assemble content)
  ↓
buildEffectiveSystemPrompt() →  SystemPrompt   (Select priority path)
  ↓
buildSystemPromptBlocks()  →  TextBlockParam[] (Chunk + cache_control tagging)
```

1. **`getSystemPrompt()`** (`src/constants/prompts.ts:444`) — Collects static segments + dynamic segments, inserts `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` marker
2. **`buildEffectiveSystemPrompt()`** (`src/utils/systemPrompt.ts:41`) — Selects by Override > Coordinator > Agent > Custom > Default priority
3. **`buildSystemPromptBlocks()`** (`src/services/api/claude.ts:3214`) — Calls `splitSysPromptPrefix()` for chunking, attaches `cache_control` to each block

## SystemPrompt Branded Type

```typescript
// src/utils/systemPromptType.ts:8
export type SystemPrompt = readonly string[] & {
  readonly __brand: 'SystemPrompt'
}
export function asSystemPrompt(value: readonly string[]): SystemPrompt {
  return value as SystemPrompt  // Zero-cost type assertion
}
```

The branded type prevents ordinary `string[]` from being accidentally passed to API calls — only explicit conversion through `asSystemPrompt()` yields the `SystemPrompt` type.

## getSystemPrompt(): Content Assembly Overview

`src/constants/prompts.ts:444` is the core factory function for the System Prompt, returning an ordered array:

| Phase | Content | Caching Strategy |
|------|------|----------|
| **Static Region** | Intro Section, System Rules, Doing Tasks, Actions, Using Tools, Tone & Style, Output Efficiency | Can be cached across organizations (`scope: 'global'`) |
| **BOUNDARY** | `SYSTEM_PROMPT_DYNAMIC_BOUNDARY = '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'` | Boundary marker (not sent to API) |
| **Dynamic Region** | Session Guidance, Memory, Model Override, Env Info, Language, Output Style, MCP Instructions, Scratchpad, FRC, Summarize Tool Results, Token Budget, Brief | Varies per session (`scope: 'org'` or uncached) |

### Dynamic Region Section Registry

The dynamic region is registered via `systemPromptSection()` / `DANGEROUS_uncachedSystemPromptSection()`, factory functions defined in `src/constants/systemPromptSections.ts`:

```typescript
// Cached Section: computed once, recomputed only after /clear or /compact
systemPromptSection('memory', () => loadMemoryPrompt())

// Dangerous: recomputed every turn, breaks Prompt Cache
DANGEROUS_uncachedSystemPromptSection(
  'mcp_instructions',
  () => isMcpInstructionsDeltaEnabled() ? null : getMcpInstructionsSection(mcpClients),
  'MCP servers connect/disconnect between turns'  // Must provide reason for cache breaking
)
```

`resolveSystemPromptSections()` resolves all Sections each query turn. For Sections with `cacheBreak: false`, the cached value from `getSystemPromptSectionCache()` is preferred. Only truly dynamic content like MCP instructions uses `DANGEROUS_uncachedSystemPromptSection`.

### `CLAUDE_CODE_SIMPLE` Fast Path

When the `CLAUDE_CODE_SIMPLE` environment variable is truthy, the entire System Prompt is reduced to a single line:

```typescript
`You are Claude Code, Anthropic's official CLI for Claude.\n\nCWD: ${getCwd()}\nDate: ${getSessionStartDate()}`
```

Skips all Section registration, cache chunking, and dynamic assembly — used for minimal token consumption in test scenarios.

## buildEffectiveSystemPrompt(): Five Priority Levels

`src/utils/systemPrompt.ts:41` determines which System Prompt is ultimately used:

| Priority | Condition | Behavior |
|--------|------|------|
| **0. Override** | `overrideSystemPrompt` is non-empty | Complete replacement, returns `[override]` |
| **1. Coordinator** | `COORDINATOR_MODE` feature + environment variable | Uses coordinator-specific prompt |
| **2. Agent** | `mainThreadAgentDefinition` exists | Proactive mode: appends to default prompt; otherwise: replaces default prompt |
| **3. Custom** | `--system-prompt` parameter specified | Replaces default prompt |
| **4. Default** | No special conditions | Uses full `getSystemPrompt()` output |

`appendSystemPrompt` is always appended to the end (except for Override).

## Caching Strategy: Chunking, Tagging, Hits

This is the most sophisticated part of System Prompt design.

### Anthropic Prompt Cache Basics

Anthropic API's Prompt Cache allows reusing the same System Prompt prefix across requests, charged at cache hit rates (much lower than full input pricing). The cache key is determined by a Blake2b hash of the content — any character change causes cache invalidation.

### `splitSysPromptPrefix()`: Three Chunking Modes

`src/utils/api.ts:321` is the core of the caching strategy, choosing between three chunking modes based on conditions:

#### Mode 1: When MCP Tools Exist (`skipGlobalCacheForSystemPrompt=true`)

```
[attribution header]    → cacheScope: null     (not cached)
[system prompt prefix]  → cacheScope: 'org'    (organization-level cache)
[everything else]       → cacheScope: 'org'    (organization-level cache)
```

MCP tool lists may change during a session (connect/disconnect), breaking the foundation for cross-organization caching, so it degrades to organization-level.

#### Mode 2: Global Cache + Boundary Present (1P Exclusive)

```
[attribution header]    → cacheScope: null     (not cached)
[system prompt prefix]  → cacheScope: null     (not cached)
[static content]        → cacheScope: 'global' (global cache! shared across organizations)
[dynamic content]       → cacheScope: null     (not cached)
```

This is the highest cache efficiency mode. Static content before `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` (Intro, Rules, Tone & Style, etc.) is identical for all users and can be cached across organizations.

#### Mode 3: Default (3P Providers or Missing Boundary)

```
[attribution header]    → cacheScope: null     (not cached)
[system prompt prefix]  → cacheScope: 'org'    (organization-level cache)
[everything else]       → cacheScope: 'org'    (organization-level cache)
```

### `getCacheControl()`: TTL Decision

`src/services/api/claude.ts:359` generates the `cache_control` object:

```typescript
{
  type: 'ephemeral',
  ttl?: '1h',         // Only when specific querySource qualifies
  scope?: 'global',   // Only for the static region
}
```

1-hour TTL determination logic (`should1hCacheTTL()`, line 394):
- **Bedrock users**: Enabled via `ENABLE_PROMPT_CACHING_1H_BEDROCK` environment variable
- **1P users**: Matched via GrowthBook-configured `allowlist` array against `querySource`, supports prefix wildcards (e.g., `"repl_main_thread*"`)
- **Session-level lock**: Eligibility determination is cached in bootstrap state, preventing mid-session GrowthBook configuration changes from causing inconsistent TTL within the same session

### Cache Breaking: Placement of Session-Specific Guidance

`getSessionSpecificGuidanceSection()` (`src/constants/prompts.ts:352`) content must be placed **after** `SYSTEM_PROMPT_DYNAMIC_BOUNDARY`. Because it contains:
- The current session's enabledTools set
- Runtime determination of `isForkSubagentEnabled()`
- Result of `getIsNonInteractiveSession()`

If these runtime bits were in the static region, they would produce 2^N Blake2b hash variants (N = number of runtime conditions), completely destroying cache hit rates. Source code comments explicitly warn:

> Each conditional here is a runtime bit that would otherwise multiply the Blake2b prefix hash variants (2^N). See PR #24490, #24171 for the same bug class.

## Context Injection: System Context and User Context

The System Prompt array itself doesn't contain runtime context (git status, CLAUDE.md content). Context is injected through two independent pipelines:

### System Context (`src/context.ts:116`)

```typescript
export const getSystemContext = memoize(async () => {
  return {
    gitStatus,           // git branch, status, recent commits (truncated to MAX_STATUS_CHARS=2000)
    cacheBreaker,        // Cache breaker for ant users only
  }
})
```

- Uses `lodash.memoize` for caching — **computed only once during the entire session**
- Git status snapshot includes 5 parallel `git` commands (branch, defaultBranch, status, log, userName)
- `status` is truncated at 2000 characters with a hint to use BashTool for more information
- When `systemPromptInjection` changes, all context caches are cleared via `getUserContext.cache.clear?.()`

### User Context (`src/context.ts:155`)

```typescript
export const getUserContext = memoize(async () => {
  return {
    claudeMd,            // Merged CLAUDE.md content
    currentDate,         // "Today's date is YYYY-MM-DD."
  }
})
```

- **CLAUDE.md disable conditions**: `CLAUDE_CODE_DISABLE_CLAUDE_MDS` environment variable, or `--bare` mode (unless directories are explicitly specified via `--add-dir`)
- `--bare` mode semantics are "skip things I didn't ask for" rather than "ignore everything"

### Injection Position

In `src/query.ts:449`:

```typescript
// System Context appended to the end of System Prompt
const fullSystemPrompt = asSystemPrompt(
  appendSystemContext(systemPrompt, systemContext)  // Simple concatenation
)
```

User Context is injected via `prependUserContext()` (`src/utils/api.ts:449`) as the first user message wrapped in `<system-reminder>` tags, placed before all conversation messages.

## Attribution Header: Billing and Security

The first block of every API request's System Prompt is the Attribution Header (`src/constants/system.ts:30`), containing:
- **`cc_version`**: Claude Code version + fingerprint
- **`cc_entrypoint`**: Entry point identifier (REPL / SDK / pipe, etc.)
- **`cch=00000`** (when NATIVE_CLIENT_ATTESTATION is enabled): Bun's native HTTP layer replaces the zeros with a computed hash value before sending; the server verifies this token to confirm the request comes from an authentic Claude Code client

The Header always has `cacheScope: null` — it varies by version and fingerprint, making it unsuitable for caching.

## CLAUDE.md: Project-Level Knowledge Injection

This is one of Claude Code's most elegant designs. Place a `CLAUDE.md` file in the project root to let AI "understand" your project:

- **Project overview**: What the project does, what tech stack it uses
- **Development conventions**: Code style, naming conventions, branching strategy
- **Common commands**: How to build, test, deploy
- **Notes**: Known pitfalls, special configurations

The system automatically discovers and merges multi-level CLAUDE.md files:

```
~/.claude/CLAUDE.md              ← User global (personal preferences)
  └── /project/CLAUDE.md         ← Project root (team shared)
        └── /project/src/CLAUDE.md  ← Subdirectory (module specific)
```

The loading logic is implemented in `src/utils/claudemd.ts` via `getClaudeMds()` and `getMemoryFiles()` — traversing the directory tree upward from CWD, merging all matching CLAUDE.md file contents.

## Design Insight: Why `string[]` Instead of a Single `string`

Designing the System Prompt as an array rather than a single text string is for **cache chunking**:

1. Anthropic Prompt Cache uses **content blocks** (TextBlock) as caching units
2. Splitting the System Prompt into multiple blocks allows unchanged parts (Intro, Rules) to achieve independent cache hits
3. If it were a single `string`, any single character change (such as a date update) would invalidate the entire System Prompt's cache
4. The `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` marker allows `splitSysPromptPrefix()` to precisely tag the static region as `scope: 'global'` and the dynamic region as untagged or `scope: 'org'`

This is Claude Code's core design for token cost optimization — a typical System Prompt is ~20K+ tokens, and cache chunking can save 30-50% of input token costs.
