## Context Window: 200K Is Not All

Claude Code's default context window is 200K tokens (`MODEL_CONTEXT_WINDOW_DEFAULT = 200_000`), but the actual space available for conversation is much smaller:

```
Context Window (200K)
├── System prompt (~15-25K, low cost after caching)
├── Tool definitions (~10-20K, including MCP tools)
├── User context (CLAUDE.md, git status, etc.)
├── Output reservation (maxOutputTokens)
│   ├── Default cap: 64K
│   ├── Actual default: 8K (slot-reservation optimization)
│   └── Auto-escalation on ceiling: one 64K retry
└── Remaining: Conversation history space (grows with conversation)
```

`getContextWindowForModel()` (`src/utils/context.ts:51`) resolves window size with 5 priority levels:

1. `CLAUDE_CODE_MAX_CONTEXT_TOKENS` environment variable override
2. Model name contains `[1m]` suffix → 1M tokens
3. `getModelCapability(model).max_input_tokens`
4. 1M beta header + supported models (claude-sonnet-4, opus-4-6)
5. Fallback: 200K

**Effective context** = window size - min(maxOutputTokens, 20K), because compaction summaries need reserved output space.

## Token Counting: Approximate vs Precise

The system uses a two-tier token counting strategy:

### Approximate Estimation (Millisecond-Level)

```typescript
// src/services/tokenEstimation.ts
function roughTokenCountEstimation(content: string, bytesPerToken = 4): number {
  return Math.round(content.length / bytesPerToken)
}
```

Special handling for different content types:
- **JSON/JSONL**: `bytesPerToken = 2` (dense `{`, `:`, `,` symbols, each only 1-2 tokens)
- **Images/Documents**: Fixed 2000 tokens (conservative estimate based on 2000x2000px upper limit)
- **thinking block**: Actual text length / 4
- **tool_use**: Serialized `name + JSON.stringify(input)` then / 4

### Precise Counting (API Call)

Uses Anthropic's `beta.messages.countTokens` endpoint. Different paths for different providers:

| Provider | Method |
|----------|------|
| Anthropic direct | `anthropic.beta.messages.countTokens()` |
| AWS Bedrock | `@aws-sdk/client-bedrock-runtime`'s `CountTokensCommand` |
| Google Vertex | Anthropic SDK + beta filtering |
| Fallback (Bedrock unsupported) | Send a `max_tokens=1` request with Haiku, read `usage.input_tokens` |

Precise counting is used at critical decision points (pre/post compaction comparison, warning determination), while approximate estimation is used in hot paths (shouldAutoCompact check each loop iteration).

## Auto-Compaction Trigger Thresholds

```
src/services/compact/autoCompact.ts — Core thresholds
```

| Constant | Value | Meaning |
|------|----|------|
| `AUTOCOMPACT_BUFFER_TOKENS` | 13,000 | Window minus this value = auto-compaction trigger point |
| `WARNING_THRESHOLD_BUFFER_TOKENS` | 20,000 | Warning displayed at trigger point + 20K |
| `ERROR_THRESHOLD_BUFFER_TOKENS` | 20,000 | Error displayed at trigger point + 20K |
| `MANUAL_COMPACT_BUFFER_TOKENS` | 3,000 | Blocking limit for manual /compact |
| `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES` | 3 | Stop attempting after 3 consecutive failures |

Using a 200K window as an example:
- **~167K**: Warning flashes, user sees a suggestion to compact
- **~180K**: Auto-compaction triggers (200K - 20K output reservation = 180K effective, minus 13K buffer)
- **~197K**: Reaches blocking limit, new messages are blocked

`shouldAutoCompact()` has multiple escape conditions:
- `compact` / `session_memory` query sources never trigger (prevents recursive deadlock)
- `DISABLE_COMPACT` / `DISABLE_AUTO_COMPACT` environment variables
- User configuration `autoCompactEnabled = false`
- Suppressed when Context Collapse mode is active (collapse manages context itself)
- Suppressed in Reactive Compact experimental mode for proactive compaction
- Exceeded consecutive failure limit (circuit breaker)

## Micro-Compact: Progressive Compression of Tool Results

Before triggering full compaction, the system first attempts **micro-compact** — only compressing old tool call results:

```
Compactable tools list (COMPACTABLE_TOOLS):
FileRead, Bash, Grep, Glob, WebSearch, WebFetch, FileEdit, FileWrite
```

Strategy based on time:
- Tool results older than a threshold (controlled by `timeBasedMCConfig`) are replaced with short placeholders
- Image/document results are replaced with `[image]` / `[document]` text
- Each replacement frees tokens, potentially deferring full compaction

Tools also have a `maxResultSizeChars` hard limit (typically 100K) — overly long results are truncated before being written to messages.

## Full Compaction Complete Flow

```
autoCompactIfNeeded() / compactConversation()
  ↓
1. Execute PreCompact hooks (external custom instructions can be injected)
  ↓
2. Attempt Session Memory compaction (lighter weight, tried first)
  ↓
3. Session Memory fails → Full compaction
   a. Strip images/documents from messages (replace with [image]/[document])
   b. Strip skill_discovery/skill_listing attachments (re-injected after compaction)
   c. Send summary request via forked agent (reuses main thread's prompt cache)
   d. If summary request itself triggers prompt-too-long → truncateHeadForPTLRetry()
      Delete from oldest API turns, retry up to 3 times
   ↓
4. After successful compaction, rebuild context:
   - compactBoundaryMarker (records compaction type, pre-compaction token count, etc.)
   - Summary message (invisible user message)
   - Re-read of 5 most recent files (POST_COMPACT_TOKEN_BUDGET = 50K)
   - plan file attachments (if any)
   - plan mode instructions (if in plan mode)
   - Invoked skill content (each skill ≤5K, total ≤25K)
   - Incremental re-injection of deferred tools / agent listing / MCP instructions
   - Re-execute SessionStart hooks
   - Execute PostCompact hooks
  ↓
5. Update cache baseline, preventing false positive cache break detection
```

### Prompt Cache Sharing

The compaction API call is one of the most expensive operations in the entire session. The system reuses the main thread's cache prefix (system prompt + tools + context messages) via `runForkedAgent`, raising cache hit rate from 2% to nearly 100%. This optimization alone saves approximately 0.76% of fleet-level `cache_creation` tokens.

## Output Token Slot Optimization

An often-overlooked optimization: **dynamic adjustment of maxOutputTokens**.

```typescript
// src/services/api/claude.ts — getMaxOutputTokensForModel()
const defaultTokens = isMaxTokensCapEnabled()
  ? Math.min(maxOutputTokens.default, 8_000)  // Default lowered to 8K
  : maxOutputTokens.default                     // Original default 32K/64K
```

Why? Because the API's slot mechanism reserves inference capacity based on `max_tokens`. BQ p99 output is only 4,911 tokens, so a 32K default wastes 8-16x slot capacity. After lowering to 8K, less than 1% of requests are truncated — these requests automatically get one clean retry at 64K.

This optimization's impact on the token budget is indirect: more slot capacity means less queue delay, indirectly reducing timeouts and retries.

## Partial Compact: Selective Compaction

Besides full compaction, users can also select a position in the message history and only compact the content before or after that position:

- **`up_to` direction**: Compacts content before the selected message, preserving recent conversation
- **`from` direction**: Compacts content after the selected message, preserving earlier conversation

The `from` direction preserves prompt cache (prefix unchanged), while the `up_to` direction breaks cache (summary is inserted before the retained content).

Both directions use the same PTL (prompt-too-long) retry strategy: delete from the oldest API turns, ensuring at least one message group is retained for summarization.
