## Compaction Trigger Timing

Context compaction is not a single operation, but a **three-tier progressive** strategy system, corresponding to different trigger conditions and severity levels:

| Tier | Trigger Condition | Implementation Location | Requires API Call |
|------|---------|---------|:---:|
| **MicroCompact** | Individual tool output too long | `microCompact.ts` | No |
| **Session Memory Compact** | Auto-compaction triggered (requires feature flag) | `sessionMemoryCompact.ts` | No |
| **Traditional API Summary** | Manual `/compact` or auto fallback when SM unavailable | `compact.ts` | Yes |

### Compaction Entry Priority Chain

Source path: `src/commands/compact/compact.ts`

When a user executes `/compact` or the system triggers auto-compaction, the compaction command tries the following priorities:

```typescript
// compact.ts:55-99 — Simplified priority chain
if (!customInstructions) {
  const sessionMemoryResult = await trySessionMemoryCompaction(messages, ...)
  if (sessionMemoryResult) return sessionMemoryResult      // Priority: SM compaction
}

if (reactiveCompact?.isReactiveOnlyMode()) {
  return await compactViaReactive(messages, ...)            // Secondary: Reactive compaction
}

// Fallback: Traditional API summary
const microcompactResult = await microcompactMessages(messages, context)
const messagesForCompact = microcompactResult.messages
// → Call AI model to generate summary
```

Note: SM compaction does not support custom instructions (e.g., `/compact focus on the auth module`); custom instructions go directly to the traditional path.

## Tier 1: MicroCompact — Local Compaction

Source path: `src/services/compact/microCompact.ts`

MicroCompact doesn't compact the entire conversation but instead **clears the content of old tool outputs**. It maintains a whitelist:

```typescript
const COMPACTABLE_TOOLS = new Set([
  'Read',      // File read
  'Bash',      // Command output
  'Grep',      // Search results
  'Glob',      // File listings
  'WebSearch', // Search results
  'WebFetch',  // Web content
  'Edit',      // Edit output
  'Write',     // Write output
])
```

Replacement strategy: Tool output content older than the time window is replaced with `[Old tool result content cleared]`. This is not simple truncation — original content remains in the JSONL transcript, just no longer sent to the API.

MicroCompact also has a **time decay configuration** (`timeBasedMCConfig.ts`): older tool outputs are more likely to be cleared, while recent ones are preserved first.

### Special Handling for Images and Documents

```typescript
const IMAGE_MAX_TOKEN_SIZE = 2000
```

Image blocks exceeding 2000 estimated tokens are also cleared by MicroCompact. PDF document blocks similarly.

## Tier 2: Session Memory Compact — No API Call Compaction

Source path: `src/services/compact/sessionMemoryCompact.ts`

When both `tengu_session_memory` and `tengu_sm_compact` feature flags are enabled, the system preferentially uses Session Memory for compaction — **no summary model call needed**, directly using already-extracted Session Memory as the conversation summary.

### Retention Window Calculation

```typescript
// sessionMemoryCompact.ts:324-397
export function calculateMessagesToKeepIndex(messages, lastSummarizedIndex) {
  const config = getSessionMemoryCompactConfig()
  // Default: minTokens=10K, minTextBlockMessages=5, maxTokens=40K

  let startIndex = lastSummarizedIndex + 1
  // Expand backward from lastSummarizedIndex until two minimums are met or the maximum is hit
  for (let i = startIndex - 1; i >= floor; i--) {
    totalTokens += estimateMessageTokens([msg])
    if (hasTextBlocks(msg)) textBlockMessageCount++
    startIndex = i
    if (totalTokens >= config.maxTokens) break
    if (totalTokens >= config.minTokens && textBlockMessageCount >= config.minTextBlockMessages) break
  }
  return adjustIndexToPreserveAPIInvariants(messages, startIndex)
}
```

This algorithm ensures the retained message window after compaction satisfies:
- At least 10,000 tokens (sufficient context depth)
- At least 5 messages containing text (conversation continuity)
- At most 40,000 tokens (not so large as to trigger the next compaction)

### Tool Pair Integrity Protection

`adjustIndexToPreserveAPIInvariants()` is a **critical correctness guarantee** in compaction:

The API requires every `tool_result` to have a corresponding `tool_use`, and vice versa. If compaction happens to cut at a `tool_result` message, it would cause an API error.

```typescript
// sessionMemoryCompact.ts:232-314
// Step 1: Scan forward, find all tool_use referenced by tool_result in retained messages
// Step 2: Scan forward, find thinking blocks sharing message.id with retained assistant messages
// Both cases require moving startIndex forward
```

Streaming splits a single assistant message into multiple storage records (thinking, tool_use, etc. each with independent uuid but sharing `message.id`), adding complexity to edge cases.

## Tier 3: Traditional API Summary Compaction

Source path: `src/services/compact/compact.ts`

When SM compaction is unavailable, the system falls back to the traditional approach: calling an AI model to generate a conversation summary.

### Pre-Compaction Processing

Before sending to the summary model, messages undergo multiple preprocessing layers:

```typescript
// compact.ts:147-202
const stripped = stripImagesFromMessages(messages)   // Images → [image] text markers
const stripped2 = stripReinjectedAttachments(stripped) // Remove attachments that will be re-injected
```

Images are replaced with `[image]` markers, preventing the summary API call itself from triggering a prompt-too-long error.

### Post-Compaction Re-Injection

After compaction, the system **re-injects key context** from the summary:

```typescript
// compact.ts:124-132
export const POST_COMPACT_TOKEN_BUDGET = 50_000          // Total budget
export const POST_COMPACT_MAX_FILES_TO_RESTORE = 5        // Restore at most 5 files
export const POST_COMPACT_MAX_TOKENS_PER_FILE = 5_000     // 5K tokens per file
export const POST_COMPACT_MAX_TOKENS_PER_SKILL = 5_000    // 5K tokens per skill
export const POST_COMPACT_SKILLS_TOKEN_BUDGET = 25_000    // Skills total budget 25K
```

This 50K token re-injection budget is used for:
1. Restoring recently read file content (up to 5 files, each truncated to 5K tokens)
2. Restoring activated skill instructions (each skill truncated to 5K tokens, 25K total)
3. Re-injecting CLAUDE.md content
4. Restoring MCP tool discovery results

## CompactBoundary: Compaction Boundary Marker

Source path: `src/utils/messages.ts` (`createCompactBoundaryMessage`)

After each compaction, the system inserts a `SystemCompactBoundaryMessage` into the message stream:

```typescript
type SystemCompactBoundaryMessage = {
  type: 'system'
  message: {
    type: 'compact_boundary'
    compactMetadata: {
      compactType: 'auto' | 'manual' | 'micro'
      preCompactTokenCount: number
      lastUserMessageUuid: string
      preCompactDiscoveredTools?: string[]
    }
  }
}
```

All subsequent operations only process messages **after the last boundary**:

```typescript
// messages.ts
export function getMessagesAfterCompactBoundary(messages: Message[]): Message[] {
  const lastBoundary = messages.findLastIndex(m => isCompactBoundaryMessage(m))
  return lastBoundary >= 0 ? messages.slice(lastBoundary + 1) : messages
}
```

### Preserved Segment Annotation

The boundary message also carries a `preservedSegment` annotation, recording which messages were preserved rather than compacted:

```typescript
// compact.ts — annotateBoundaryWithPreservedSegment
boundaryMarker.compactMetadata.preservedSegment = {
  summaryMessageUuid: string
  preservedMessageUuids: string[]
}
```

This helps the loader correctly rebuild the message chain during session recovery, avoiding re-compacting already preserved messages.

## PTL Emergency Degradation: Prompt Too Long

When compaction still exceeds the token limit (`PROMPT_TOO_LONG` error), the system enters the emergency degradation path:

1. **Reactive Compact**: `reactiveCompactOnPromptTooLong()` attempts more aggressive compaction
2. **Truncation retry**: If reactive also fails, `truncateHeadForPTLRetry()` directly truncates the earliest messages
3. Give up and report error

Reactive Compact is currently a stub in the decompiled version (`isReactiveOnlyMode() → false`), indicating this is an experimental feature internal to Anthropic.

## Compaction Hook Mechanism

Custom Hooks can be executed before and after compaction:

- **Pre-compact Hook** (`executePreCompactHooks`): Executes before compaction, can inject "must preserve" markers
- **Post-compact Hook** (`executePostCompactHooks`): Executes after compaction, can verify whether key information was preserved
- **Session Start Hook** (`processSessionStartHooks('compact')`): Used by SM compaction to restore context like CLAUDE.md

Hook results are attached to compaction results as `HookResultMessage`, ensuring user custom logic is respected during the compaction process.
