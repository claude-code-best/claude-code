## Memory System Storage Architecture

Source path: `src/memdir/paths.ts`, `src/memdir/memdir.ts`

Claude Code's memory system is **pure files** — no database, no vector store, just Markdown files and directory structure.

### Directory Layout

```
~/.claude/projects/<sanitized-git-root>/memory/
├── MEMORY.md                    ← Entry index (loaded every conversation)
├── user_role.md                 ← User memory
├── feedback_testing.md          ← Feedback memory
├── project_mobile_release.md    ← Project memory
├── reference_linear_ingest.md   ← Reference memory
└── logs/                        ← KAIROS mode: daily logs
    └── 2026/
        └── 04/
            └── 2026-04-01.md
```

Path resolution chain (`getAutoMemPath()`):
1. `CLAUDE_COWORK_MEMORY_PATH_OVERRIDE` environment variable (Cowork SDK full path override)
2. `autoMemoryDirectory` setting (limited to `policySettings`/`localSettings`/`userSettings` — **intentionally excludes** `projectSettings` to prevent a malicious repository from pointing the memory path to `~/.ssh`)
3. Default: `<memoryBase>/projects/<sanitized-git-root>/memory/`

All worktrees of the same Git repository share one memory directory (via `findCanonicalGitRoot()` to find the real `.git` root).

### MEMORY.md Index

`MEMORY.md` is the entry index for memory, loaded in full into the context every conversation:

```typescript
// memdir.ts:35-38
export const ENTRYPOINT_NAME = 'MEMORY.md'
export const MAX_ENTRYPOINT_LINES = 200
export const MAX_ENTRYPOINT_BYTES = 25_000
```

The index has **dual limits**: 200 lines AND 25KB. Exceeding either triggers `truncateEntrypointContent()` with a warning appended. Design reasoning: p97 of index files are covered by 200 lines, but some index entries are exceptionally long (p100 observed 197KB/200 lines), so the byte limit catches this long-line anomaly.

Index entry format:
```markdown
- [Title](file.md) — one-line hook
```

One entry per line, ~150 characters or fewer. `MEMORY.md` itself has no frontmatter — it is just a link list, not memory content.

## Four-Type Taxonomy

Source path: `src/memdir/memoryTypes.ts`

Memories are constrained to a **closed four-type system**, each type with explicit `<when_to_save>`, `<how_to_use>`, and `<body_structure>` specifications:

| Type | Stored Content | Typical Trigger |
|------|---------|---------|
| **user** | User role, preferences, technical background | "I'm a data scientist", "I've written Go for ten years" |
| **feedback** | User corrections and confirmations of AI behavior | "Don't mock the database", "Single PR is better" |
| **project** | Project context not derivable from code | "Merge freeze starts Thursday", "Auth rewrite is a compliance requirement" |
| **reference** | External system pointers | "Pipeline bugs are in the Linear INGEST project" |

Key design constraint: **only store information that cannot be derived from the current project state**. Code architecture, file paths, and git history can all be obtained in real time and don't need memorization.

### Feedback Type Dual-Channel Capture

The `feedback` type's `when_to_save` instruction specifically emphasizes:

> Record from failure AND success: if you only save corrections, you will avoid past mistakes but drift away from approaches the user has already validated, and may grow overly cautious.

This means AI saves not only when the user says "don't do that" but also when the user says "yes, exactly like that." The latter is harder to capture but equally important — it prevents AI behavior from drifting over time.

### Frontmatter Format for Each Memory

```markdown
---
name: {{memory name}}
description: {{one-line description — used for future relevance judgment}}
type: {{user, feedback, project, reference}}
---

{{memory content — feedback/project types recommend including **Why:** and **How to apply:** lines}}
```

The `description` field is key: it's not a human-readable summary, but search keywords for the AI recall system to make relevance judgments.

## Intelligent Recall Mechanism

Source path: `src/memdir/findRelevantMemories.ts`, `src/memdir/memoryScan.ts`

Not all memories are appropriate for every conversation. The system uses a **lightweight Sonnet side query** to filter the most relevant memories.

### Recall Flow

```
User message → findRelevantMemories(query, memoryDir)
  ├── scanMemoryFiles() — Scan all memory file frontmatter
  ├── selectRelevantMemories() — Sonnet side query, select ≤5 from the manifest
  └── Return [{path, mtimeMs}, ...]
```

The core is the `selectRelevantMemories()` function, which calls `sideQuery()` (an independent lightweight API call):

```typescript
// findRelevantMemories.ts:98-121
const result = await sideQuery({
  model: getDefaultSonnetModel(),  // Use Sonnet for filtering (not the main model)
  system: SELECT_MEMORIES_SYSTEM_PROMPT,
  messages: [{
    role: 'user',
    content: `Query: ${query}\n\nAvailable memories:\n${manifest}${toolsSection}`
  }],
  max_tokens: 256,
  output_format: { type: 'json_schema', schema: { ... } },
})
```

### Recent Tool Denoising

When AI is currently using a tool, recalling usage documentation for that tool is noise (working context already exists in the conversation). The `recentTools` parameter lets the recall system skip these memories:

```typescript
// findRelevantMemories.ts:92-95
const toolsSection = recentTools.length > 0
  ? `\n\nRecently used tools: ${recentTools.join(', ')}`
  : ''
```

The System Prompt explicitly instructs: "If a recently used tools list is provided, do not select usage references or API documentation for those tools. **Still select** warnings, pitfalls, or known issues about these tools — that's the most critical information during use."

### Already-Surfaced Deduplication

The `alreadySurfaced` parameter filters file paths already shown in previous turns, letting Sonnet's 5-slot budget be spent on new candidates rather than re-recalling the same file.

## Memory Injection into System Prompt Pipeline

Source path: `src/memdir/memdir.ts` → `src/context.ts`

`loadMemoryPrompt()` is the entry point for memory injection, called once per session (cached via `systemPromptSection('memory', ...)`):

```typescript
// memdir.ts:419-507
export async function loadMemoryPrompt(): Promise<string | null> {
  // Priority: KAIROS log mode → TEAMMEM combined mode → pure auto-memory
  if (feature('KAIROS') && autoEnabled && getKairosActive()) {
    return buildAssistantDailyLogPrompt(skipIndex)
  }
  if (feature('TEAMMEM') && teamMemPaths!.isTeamMemoryEnabled()) {
    return teamMemPrompts!.buildCombinedMemoryPrompt(...)
  }
  if (autoEnabled) {
    return buildMemoryLines('auto memory', autoDir, ...).join('\n')
  }
  return null
}
```

Injection timing: When `getSystemContext()` is called in `context.ts`, the memory Prompt is assembled as one section of the system prompt. `MEMORY.md` content is injected as a **user context message** (not system prompt), enabling Prompt Cache prefix sharing.

## KAIROS Mode: Daily Logs

Source path: `src/memdir/memdir.ts` (`buildAssistantDailyLogPrompt`)

Long-running assistant sessions use a different memory strategy:

- **Standard mode**: AI maintains `MEMORY.md` as a live index + independent memory files
- **KAIROS mode**: AI only appends logs to date files (`logs/YYYY/MM/YYYY-MM-DD.md`), no reorganization

```typescript
// Log path pattern (not a literal path — because the Prompt is cached)
const logPathPattern = join(memoryDir, 'logs', 'YYYY', 'MM', 'YYYY-MM-DD.md')
```

A separate nightly `/dream` skill is responsible for distilling logs into topic files + `MEMORY.md` index.

## Memory Drift Defense

Source path: `src/memdir/memoryTypes.ts` (`TRUSTING_RECALL_SECTION`)

Memories can become stale. The system sets up a dedicated section in the Prompt called "Before recommending from memory":

```
A memory that names a specific function, file, or flag is a claim
that it existed *when the memory was written*. It may have been
renamed, removed, or never merged. Before recommending it:

- If the memory names a file path: check the file exists.
- If the memory names a function or flag: grep for it.
```

This section's title was validated through A/B testing: "Before recommending from memory" (action-oriented) performed better than "Trusting what you recall" (abstract description) at 3/3 vs 0/3.

### Strict Semantics for Ignoring Memory

```
If the user says to *ignore* or *not use* memory:
proceed as if MEMORY.md were empty.
Do not apply remembered facts, cite, compare against,
or mention memory content.
```

This addresses a common AI anti-pattern: the user says "ignore the memory about X," and AI correctly identifies the code but still adds "unlike what memory said about Y" — that's not "ignoring," that's "acknowledging then overriding."

## Session Memory and Compaction Integration

Source path: `src/services/compact/sessionMemoryCompact.ts`

The memory system has deep integration with context compaction. When both `tengu_session_memory` and `tengu_sm_compact` feature flags are enabled, compaction preferentially uses Session Memory over traditional summarization:

```typescript
// sessionMemoryCompact.ts:57-61
const DEFAULT_SM_COMPACT_CONFIG = {
  minTokens: 10_000,           // Retain at least 10K tokens after compaction
  minTextBlockMessages: 5,     // Retain at least 5 text messages
  maxTokens: 40_000,           // Retain at most 40K tokens
}
```

SM-compact doesn't call the compaction API (no summary model) — it directly uses existing Session Memory as the summary — faster, cheaper, and without information loss.
