## Why Runtime A/B Testing Is Needed

Build-time `feature()` is "all-or-nothing" — either all users have it, or none do. But product teams need finer-grained control:

- Gradually roll out a new feature to only 5% of users
- Differentiate experiences by subscription type (Free / Pro / Team)
- Silently enable experimental capabilities for specific organizations
- Remotely disable a problematic feature at any time without shipping a new release

This is where **GrowthBook** comes in — a runtime, user-attribute-based feature gating and A/B testing system.

## Integration Architecture

GrowthBook's complete implementation lives in `src/services/analytics/growthbook.ts` (1156 lines), with the following workflow:

<Steps>
  <Step title="Fetch Remote Configuration at Startup">
    At CLI startup, the GrowthBook SDK fetches current feature configuration and experiment group rules via the `https://api.anthropic.com/` API endpoint. It uses `remoteEval: true` mode — group assignment is computed server-side, and the client only receives the results.
  </Step>
  <Step title="Compute User Attributes">
    The SDK collects current user attributes (device ID, subscription type, organization UUID, etc.) to determine which experiment groups the user belongs to.
  </Step>
  <Step title="Cache Locally">
    Results are cached in the `cachedGrowthBookFeatures` field of `~/.claude.json`. Refresh interval: 20 minutes for Anthropic employees, 6 hours for external users.
  </Step>
  <Step title="Query Flags in Code">
    Business code queries feature status via `tengu_*`-prefixed flag names, and the GrowthBook SDK returns the current user's group assignment value.
  </Step>
</Steps>

## User Targeting Attributes

GrowthBook determines experiment group assignment based on the following user attributes:

| Attribute | Type | Source | Purpose |
|------|------|------|------|
| `id` | string | Session ID | Per-session grouping |
| `deviceID` | string | Persistent device identifier | Cross-session consistency |
| `sessionId` | string | Current session ID | Session-level experiments |
| `platform` | enum | `process.platform` | OS-based differentiation |
| `organizationUUID` | string | API auth info | Per-organization rollout |
| `accountUUID` | string | API auth info | Per-account rollout |
| `subscriptionType` | string | API auth info | Free / Pro / Team differentiation |
| `rateLimitTier` | string | API auth info | By rate limit tier |
| `email` | string | API auth info | Precise targeting of specific users |
| `appVersion` | string | `MACRO.VERSION` | Version-based rollout |
| `github` | object | GitHub Actions metadata | CI environment special handling |

<Note>
This targeting system means Anthropic can run very fine-grained experiments — for example, "enable a new feature for only 10% of Pro subscribers on Mac."
</Note>

## Codename Culture: The World of tengu_*

All runtime flags use the `tengu_` prefix — "Tengu" is Claude Code's internal project codename. Flag names use an **animal/plant/mineral + adjective** naming convention, intentionally kept opaque.

<AccordionGroup>
  <Accordion title="tengu_kairos — Kairos Assistant Mode">
    Controls the runtime switch for the KAIROS feature. Even if build-time `feature('KAIROS')` passes, this flag must also be hit for activation. Dual gating ensures new features can be rolled out in stages.
  </Accordion>
  <Accordion title="tengu_amber_stoat — Explore Agent A/B Test">
    Controls behavior variants of the built-in Explore sub-agent. "amber stoat" is a randomly generated codename unrelated to the feature content — this is to prevent guessing functionality from flag names.
  </Accordion>
  <Accordion title="tengu_auto_background_agents — Background Agent Automation">
    Controls whether certain tasks are automatically dispatched to background agents for execution, rather than blocking the user in the foreground.
  </Accordion>
  <Accordion title="tengu_onyx_plover — Auto-Dream Background Memory">
    Controls the "auto-dream" feature — organizing and consolidating agent memory in the background during idle time. "onyx plover" is another opaque codename.
  </Accordion>
  <Accordion title="tengu_glacier_2xr — Tool Search Behavior">
    Controls behavior variants of Tool Search, possibly an A/B test of search algorithms or ranking strategies.
  </Accordion>
  <Accordion title="tengu_birch_trellis — Bash Permission Policy">
    Controls policy variants for BashTool permission decisions — possibly testing more permissive or more restrictive permission rules.
  </Accordion>
  <Accordion title="tengu_scratch — Scratchpad Feature">
    Controls an experimental "scratchpad" feature, possibly allowing AI to use an intermediate staging area when handling complex tasks.
  </Accordion>
  <Accordion title="tengu_quartz_lantern — Diff Calculation Strategy">
    Controls the diff calculation method for file writes and edits. Possibly A/B testing the impact of different diff algorithms on user experience.
  </Accordion>
</AccordionGroup>

## Ant-Only Override Mechanism

Anthropic employees have two ways to bypass GrowthBook's remote evaluation:

### Environment Variable Override

```bash
# Only effective in USER_TYPE=ant builds
CLAUDE_INTERNAL_FC_OVERRIDES='{"tengu_kairos": true}' claude
```

Pass a JSON object via the `CLAUDE_INTERNAL_FC_OVERRIDES` environment variable to directly override any flag's value.

### Config UI Override

In internal builds, the `/config` command's Gates tab provides a graphical flag management interface for toggling any GrowthBook flag in real time.

## Experiment Tracking

GrowthBook integrates complete experiment exposure tracking:

- Records experiment exposure events each time a flag is queried
- Reports via protobuf-formatted `GrowthbookExperimentEvent`
- Includes `variation_id` (0=control group, 1+=experiment groups) and `in_experiment` marker
- Data is used to analyze the causal impact of features on user behavior

<Note>
GrowthBook is being migrated from Statsig — the codebase still retains compatibility layers like `checkStatsigFeatureGate_CACHED_MAY_BE_STALE()`.
</Note>
