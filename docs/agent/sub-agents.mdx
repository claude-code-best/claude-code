## Two Sub-Agent Paths: Named Agent vs Fork

`AgentTool.call()` takes two completely different paths based on whether `subagent_type` is provided (`AgentTool.tsx:322-356`):

| Dimension | Named Agent (`subagent_type` specified) | Fork Subprocess (`subagent_type` omitted) |
|------|-------------------------------------|--------------------------------------|
| **Trigger Condition** | `subagent_type` has a value | `isForkSubagentEnabled()` && type not specified |
| **System Prompt** | Agent's own `getSystemPrompt()` | Inherits parent Agent's full System Prompt |
| **Tool Pool** | `assembleToolPool()` independently assembled | Parent Agent's original tool pool (`useExactTools: true`) |
| **Context** | Task description only | Parent Agent's full conversation history (`forkContextMessages`) |
| **Model** | Can be independently specified | Inherits parent model (`model: 'inherit'`) |
| **Permission Mode** | Agent definition's `permissionMode` | `'bubble'` (bubbles up to parent terminal) |
| **Purpose** | Specialized task delegation | Prompt Cache hit rate optimization |

The core design of the Fork path is **Prompt Cache sharing**: all fork subprocesses share the parent Agent's complete `assistant` messages (all `tool_use` blocks), filled with identical placeholder `tool_result`s, with only the last `text` block containing their respective instructions. This makes API request prefix bytes identical, maximizing cache hits.

```typescript
// forkSubagent.ts:142 ŌĆö Placeholder result for all fork subprocesses
const FORK_PLACEHOLDER_RESULT = 'Fork started ŌĆö processing in background'

// buildForkedMessages() builds:
// [assistant(full tool_use), user(placeholder_results..., subprocess instruction)]
```

### Fork Recursion Prevention

Fork subprocesses retain the Agent tool (for cache-identical tool defs), but prevent recursive forking through two defenses (`AgentTool.tsx:332`):

1. **`querySource` check** (compaction safe): `context.options.querySource === 'agent:builtin:fork'`
2. **Message scan** (degraded fallback): Detect `<fork-boilerplate>` tags

## Independent Tool Pool Assembly

Sub-agents don't inherit the parent Agent's tool restrictions ŌĆö their tool pool is assembled completely independently (`AgentTool.tsx:573-577`):

```typescript
const workerPermissionContext = {
  ...appState.toolPermissionContext,
  mode: selectedAgent.permissionMode ?? 'acceptEdits'
}
const workerTools = assembleToolPool(workerPermissionContext, appState.mcp.tools)
```

Key design decisions:
- **Independent permission mode**: Sub-agents use `selectedAgent.permissionMode` (default `acceptEdits`), not restricted by the parent Agent's current mode
- **MCP tool inheritance**: `appState.mcp.tools` contains all connected MCP tools; sub-agents automatically get them
- **Agent-level MCP servers**: `initializeAgentMcpServers()` in `runAgent()` can additionally connect dedicated MCP servers for specific Agents

### Tool Filtering via resolveAgentTools

`runAgent.ts:500-502` further filters after tool assembly:

```typescript
const resolvedTools = useExactTools
  ? availableTools                           // Fork: Use parent tools directly
  : resolveAgentTools(agentDefinition, availableTools, isAsync).resolvedTools
```

`resolveAgentTools()` filters available tools based on the `tools` field in the Agent definition, mapping `['*']` to the full tool set.

## Worktree Isolation Mechanism

The `isolation: "worktree"` parameter makes sub-agents work in an independent git worktree (`AgentTool.tsx:590-593`):

```typescript
const slug = `agent-${earlyAgentId.slice(0, 8)}`
worktreeInfo = await createAgentWorktree(slug)
```

Worktree lifecycle:
1. **Creation**: Creates an independent working copy under `.git/worktrees/`
2. **CWD override**: `runWithCwdOverride(worktreePath, fn)` makes all file operations execute in the worktree
3. **Path translation**: Fork + worktree injects a path translation notice (`buildWorktreeNotice`)
4. **Cleanup** (`cleanupWorktreeIfNeeded`):
   - Hook-based worktree ŌåÆ Always preserved
   - Has changes ŌåÆ Preserved, returns `worktreePath`
   - No changes ŌåÆ Automatically deleted

## Lifecycle Management: Sync vs Async

### Async Agent (Background Running)

When `run_in_background=true` or `selectedAgent.background=true`, the Agent immediately returns an `async_launched` status (`AgentTool.tsx:686-764`):

```
registerAsyncAgent(agentId, ...)      ŌåÉ Register to AppState.tasks
  Ōåō (void ŌĆö fire and forget)
runAsyncAgentLifecycle()              ŌåÉ Background execution
  Ōö£ŌöĆŌöĆ runAgent().onCacheSafeParams    ŌåÉ Progress summary initialization
  Ōö£ŌöĆŌöĆ Message stream iteration
  Ōö£ŌöĆŌöĆ completeAsyncAgent()            ŌåÉ Mark complete
  Ōö£ŌöĆŌöĆ classifyHandoffIfNeeded()       ŌåÉ Safety check
  ŌööŌöĆŌöĆ enqueueAgentNotification()      ŌåÉ Notify main Agent
```

Async Agents get an independent `AbortController`, not shared with the parent Agent ŌĆö pressing ESC to cancel the main thread won't kill background Agents.

### Sync Agent (Foreground Running)

The key feature of sync Agents is **backgroundability** (`AgentTool.tsx:818-833`):

```typescript
const registration = registerAgentForeground({
  autoBackgroundMs: getAutoBackgroundMs() || undefined  // Default 120s
})
backgroundPromise = registration.backgroundSignal.then(...)
```

In each agentic loop iteration, the system uses `Promise.race` to race between the next message and the backgrounding signal:

```typescript
const raceResult = await Promise.race([
  nextMessagePromise.then(r => ({ type: 'message', result: r })),
  backgroundPromise  // Triggers after autoBackgroundMs
])
```

After backgrounding, the foreground iterator is terminated (`agentIterator.return()`), and a new `runAgent()` restarts with `isAsync: true`, continuing to write to the foreground's output file.

## Result Return Format

`mapToolResultToToolResultBlockParam()` returns different formats based on status (`AgentTool.tsx:1298-1375`):

| Status | Return Content |
|------|---------|
| `completed` | Content + `<usage>` block (token/tool_calls/duration) |
| `async_launched` | agentId + outputFile path + operation instructions |
| `teammate_spawned` | agent_id + name + team_name |
| `remote_launched` | taskId + sessionUrl + outputFile |

For one-off built-in Agents (Explore, Plan), the `<usage>` block is omitted ŌĆö saving approximately 1-2 Gtok of context window per week.

## MCP Dependency Wait Mechanism

If an Agent declares `requiredMcpServers`, `call()` waits for these servers to finish connecting (`AgentTool.tsx:371-410`):

```typescript
const MAX_WAIT_MS = 30_000     // Wait up to 30 seconds
const POLL_INTERVAL_MS = 500   // Poll every 500ms
```

Early exit condition: Waiting stops immediately when any required server enters a `failed` state. Tool availability is determined by resolving `mcp__` prefixed tool names (`mcp__serverName__toolName`).
