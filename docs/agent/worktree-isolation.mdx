## Why File-Level Isolation Is Needed

When multiple Agents work in parallel, sharing the same working directory causes three types of conflicts:

1. **Write conflicts**: Two Agents edit `config.ts` simultaneously, the later write overwrites the earlier one
2. **State interference**: Agent A's tests depend on a certain environment state, Agent B's modifications break it
3. **Indistinguishable**: Half-completed modifications are mixed together, making it impossible to tell which are from which Agent

Git worktree is git's native solution — creating multiple independent working directories within the same repository, each on its own branch.

## Directory Structure and Naming Rules

Worktree files are uniformly stored under `.claude/worktrees/` in the repository root:

```
<repo-root>/
├── .claude/
│   └── worktrees/
│       ├── fix-auth-bug/          # worktree working directory
│       │   ├── .git               # Link file pointing to main repository
│       │   └── src/...            # Independent filesystem view
│       └── add-dark-mode/         # Another worktree
│           └── ...
├── src/                           # Main working directory (unaffected)
└── .git/                          # Main repository
```

Branch naming follows the pattern `worktree/<slug>`, where slug is validated by `validateWorktreeSlug()`: each `/`-separated segment only allows letters, digits, `.`, `_`, `-`, with total length <= 64 characters. When unspecified, it is auto-generated using the plan slug.

## Creation Flow: EnterWorktreeTool

`EnterWorktreeTool` (`src/tools/EnterWorktreeTool/EnterWorktreeTool.ts`) execution pipeline:

```
EnterWorktreeTool.call({ name? })
  ↓
1. Check if already in a worktree (prevent nesting)
  ↓
2. Resolve to main repository root (findCanonicalGitRoot)
   If currently inside a worktree, chdir to main repository
  ↓
3. Generate slug (user-provided or plan slug)
  ↓
4. createWorktreeForSession(sessionId, slug)
   ├── Has WorktreeCreate hook?
   │   └── Execute hook, return hook-specified path (supports non-git VCS)
   └── No hook → git native path:
       a. getOrCreateWorktree(repoRoot, slug)
          ├── Fast recovery: Check if worktree directory already exists
          │   └── Read HEAD SHA from .git pointer file (no subprocess)
          └── New creation:
              i.   mkdir .claude/worktrees/ (recursive)
              ii.  fetch origin/<default-branch> (skipped if cached)
              iii. git worktree add -b worktree/<slug> <path> <base>
              iv.  performPostCreationSetup() (sparse checkout, etc.)
  ↓
5. Update process state:
   - process.chdir(worktreePath)
   - setCwd(worktreePath)
   - setOriginalCwd(worktreePath)
   - saveWorktreeState(session) → Persist to project configuration
   - clearSystemPromptSections() → Recompute cwd info in system prompt
   - clearMemoryFileCaches() → Reload CLAUDE.md in the worktree
  ↓
6. Return worktreePath and worktreeBranch
```

### Hook-First Architecture

`createWorktreeForSession()` first checks `hasWorktreeCreateHook()` — if the user has configured a `WorktreeCreate` hook in settings.json, the system doesn't call git at all, but executes the hook command and uses the returned path as the worktree path. This allows non-git version control systems (such as Pijul, Mercurial) to integrate via hooks.

### Fast Recovery Path

`getOrCreateWorktree()` has a key optimization: if the target path already exists, it reads the HEAD SHA directly from the `.git` pointer file (pure file I/O, no subprocess), skipping the entire `fetch` + `worktree add` flow. In large repositories where `fetch` takes 6-8 seconds, this optimization reduces recovery scenario latency to near zero.

## Exit Flow: ExitWorktreeTool

`ExitWorktreeTool` (`src/tools/ExitWorktreeTool/ExitWorktreeTool.ts`) supports two exit strategies:

### keep: Preserve Worktree

```
keepWorktree()
  ↓
1. chdir back to originalCwd
2. Clear currentWorktreeSession
3. Update project configuration (activeWorktreeSession = undefined)
4. Worktree directory and branch remain on disk
```

Users can continue working via `cd <worktreePath>`, or manually merge later.

### remove: Delete Worktree

With strict **safety protections**:

```
validateInput() — First line of defense
  ↓
1. Check if in a session created by EnterWorktree
   (Manually created worktrees will not be deleted)
  ↓
2. countWorktreeChanges(worktreePath, originalHeadCommit)
   ├── git status --porcelain → Count uncommitted files
   ├── git rev-list --count <originalHead>..HEAD → Count new commits
   └── Return null (if git fails) → fail-closed (refuse to delete)
  ↓
3. Has uncommitted files or new commits?
   → Refuse, require discard_changes: true confirmation
```

```
call() — Actual execution
  ↓
1. Recount changes (new modifications may have occurred between validateInput and call)
2. If tmux session exists → killTmuxSession()
3. cleanupWorktree()
   ├── hook-based → Execute WorktreeRemove hook
   └── git-based → git worktree remove --force + git branch -D
4. restoreSessionToOriginalCwd()
   - setCwd(originalCwd)
   - setOriginalCwd(originalCwd)
   - Only restore if projectRoot is the worktree (prevent accidental trigger)
   - Update hooks config snapshot
   - Clear system prompt and memory caches
```

### fail-closed Design

`countWorktreeChanges()` returns `null` ("unknown, assume unsafe") in these cases:
- `git status` or `git rev-list` exits non-zero (lock files, corrupted index)
- `originalHeadCommit` is undefined (hook-based worktree didn't set a baseline commit)

When `null` is returned, `validateInput` refuses deletion — preferring to let the user handle it manually rather than risk losing work.

## Integration with Agent Tools

The Agent tool (`AgentTool`)'s `isolation` parameter determines whether a sub-agent runs in a worktree:

- `isolation: "worktree"` → Calls `createWorktreeForSession()`, sub-agent executes in an independent worktree
- No isolation → Sub-agent shares the main working directory

Handling when a sub-agent ends:
- **Success**: Main Agent preserves the worktree via `ExitWorktreeTool(action: "keep")`, then manually merges
- **Failure/Abandon**: Main Agent cleans up via `ExitWorktreeTool(action: "remove", discard_changes: true)`

## Session State Persistence

The `WorktreeSession` object is persisted to disk via `saveCurrentProjectConfig()`, containing:

```typescript
{
  originalCwd: string,         // Working directory before entering worktree
  worktreePath: string,        // Absolute path of the worktree
  worktreeName: string,        // slug
  worktreeBranch?: string,     // Branch name (e.g., worktree/fix-auth)
  originalBranch?: string,     // Branch before entry
  originalHeadCommit?: string, // HEAD commit before entry (for change statistics)
  sessionId: string,           // Session ID that created this worktree
  tmuxSessionName?: string,    // Associated tmux session
  hookBased?: boolean,         // Whether created by hook
  creationDurationMs?: number, // Creation duration (for analysis)
}
```

This enables correct worktree context restoration during session recovery (`--resume`) — even if the process restarts, `getCurrentWorktreeSession()` reads state from project configuration.

## Sparse Checkout Optimization

For large monorepos, worktrees support `sparsePaths` configuration — checking out only specific directories rather than the entire repository. In a 210K-file repository, this reduces worktree creation time from tens of seconds to just a few seconds.

Configuration is at `getInitialSettings().worktree?.sparsePaths`, applied in `performPostCreationSetup()`.
