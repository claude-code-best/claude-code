## Architectural Differences Between Two Collaboration Modes

| Dimension | Coordinator Mode | Agent Swarms |
|------|-----------------|--------------|
| **Gating** | `feature('COORDINATOR_MODE')` + `CLAUDE_CODE_COORDINATOR_MODE=1` | Task System V2 (enabled by default) |
| **Topology** | Star: Coordinator in the center, Workers on the periphery | Mesh: Peer Agents sharing a task list |
| **Roles** | Clear division: Coordinator orchestrates, Workers execute | Blurred: Each Agent autonomously claims tasks |
| **Communication** | `SendMessage` directed communication + `<task-notification>` | Task filesystem + mailbox broadcast |
| **Use Case** | Complex tasks requiring centralized decisions | Highly parallelizable independent subtasks |

The two are not mutually exclusive — Coordinator Mode can run on top of the Swarm architecture, with the Coordinator as a special Leader Agent.

## Coordinator Mode: Star Orchestration Architecture

### Activation Mechanism

```typescript
// src/coordinator/coordinatorMode.ts:36
export function isCoordinatorMode(): boolean {
  if (feature('COORDINATOR_MODE')) {
    return isEnvTruthy(process.env.CLAUDE_CODE_COORDINATOR_MODE)
  }
  return false  // Always false in external builds
}
```

Coordinator Mode requires dual gating: build-time `feature('COORDINATOR_MODE')` and runtime environment variable. `matchSessionMode()` automatically syncs mode state during session recovery — if the recovered session is in coordinator mode, it flips the environment variable to ensure consistency.

### Coordinator's Tool Set

The Coordinator is stripped of all "hands-on" tools, retaining only orchestration capabilities:

| Tool | Purpose |
|------|------|
| **Agent** | Start new Workers (`subagent_type: "worker"`) |
| **SendMessage** | Send follow-up instructions to existing Workers |
| **TaskStop** | Stop Workers going in the wrong direction mid-task |
| **subscribe_pr_activity** | Subscribe to GitHub PR events (review comments, CI results) |

The Coordinator **does not write code, read files, or execute commands** — it does only three things: understand requirements, assign tasks, synthesize results.

### Worker Tool Permissions

Worker available tools are dynamically injected into the System Prompt by `getCoordinatorUserContext()` (`coordinatorMode.ts:80`):

```typescript
// Simplified mode: Only Bash + Read + Edit
const workerTools = isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE')
  ? [BASH_TOOL_NAME, FILE_READ_TOOL_NAME, FILE_EDIT_TOOL_NAME]
  : Array.from(ASYNC_AGENT_ALLOWED_TOOLS)
      .filter(name => !INTERNAL_WORKER_TOOLS.has(name))
```

`INTERNAL_WORKER_TOOLS` (TeamCreate, TeamDelete, SendMessage, SyntheticOutput) are explicitly excluded — Workers cannot nest team creation or send messages, preventing uncontrollable recursion.

### Scratchpad: Shared Knowledge Base Across Workers

When the `tengu_scratch` feature flag is enabled, the Coordinator has a Scratchpad directory:

```
Scratchpad directory:
  - Workers can freely read/write without permission approval
  - Used for persistent cross-Worker knowledge
  - Structure is determined by the Coordinator (no fixed format)
```

This is a critical collaboration primitive — Worker A's research results can be written to the Scratchpad, and Worker B reads them directly, without relay through the Coordinator.

### `<task-notification>` Communication Protocol

When a Worker completes, the Coordinator receives an XML-formatted notification:

```xml
<task-notification>
  <task-id>agent-a1b</task-id>          ← Worker's agentId
  <status>completed|failed|killed</status>
  <summary>Agent "Investigate auth bug" completed</summary>
  <result>Found null pointer in src/auth/validate.ts:42...</result>
  <usage>
    <total_tokens>N</total_tokens>
    <tool_uses>N</tool_uses>
    <duration_ms>N</duration_ms>
  </usage>
</task-notification>
```

Notifications are delivered as `user-role messages`, and the Coordinator distinguishes them from user messages via the `<task-notification>` tag. `<task-id>` is used for the `SendMessage` `to` parameter, enabling directed follow-up communication.

### Coordinator's Core Responsibility: Synthesis

The Coordinator System Prompt (`coordinatorMode.ts:111-369`, approximately 260 lines) explicitly requires the Coordinator **not to lazily delegate understanding**:

```
Anti-pattern (prohibited):
  "Based on your findings, fix the auth bug"
  → Pushes the responsibility of understanding to the Worker

Correct approach:
  "Fix the null pointer in src/auth/validate.ts:42.
   The user field on Session (src/auth/types.ts:15) is
   undefined when sessions expire but the token remains cached.
   Add a null check before user.id access."
  → Coordinator understood the problem itself, gives precise instructions
```

This is the most core design constraint of Coordinator Mode: the Coordinator must understand first, then assign.

## Agent Swarms: Swarm-Style Collaboration

Swarm mode is based on Task System V2 (see [Task Management](../tools/task-management.mdx)), with the core mechanism being **shared task list + competitive claiming**:

### Team Initialization

```
Leader creates team (TeamCreateTool)
  ↓
Set teamName → setLeaderTeamName()
  ↓
All teammates automatically get the same taskListId
  ↓
When teammate starts:
  1. CLAUDE_CODE_TASK_LIST_ID environment variable (explicit override)
  2. teammate context's teamName (shares leader's task list)
  3. CLAUDE_CODE_TEAM_NAME environment variable
  4. teamName set by leader
  5. getSessionId() (fallback)
```

Multi-level priority ensures the Leader and all Teammates point to the same task list without additional coordination.

### Task Claiming and Competition

`claimTask()` is the Swarm's core concurrency primitive:

```
Teammate A calls TaskList → Finds task #3 is pending
Teammate B simultaneously finds task #3 is pending
  ↓
Both simultaneously attempt TaskUpdate(task #3, {status: "in_progress"})
  ↓
File lock + high-water mark guarantees atomicity:
  - First writer gets owner lock
  - Second writer receives already_claimed error
  ↓
Teammate that got the task performs the work
  ↓
Upon completion TaskUpdate(task #3, {status: "completed"})
  → Tasks depending on this one are automatically unblocked
  → tool_result prompts "Call TaskList to find your next task"
```

### Teammate Lifecycle Management

```
Teammate exits abnormally
  ↓
unassignTeammateTasks()
  → Scan task list, find incomplete tasks where owner === teammateName
  → Reset to pending + owner=undefined
  ↓
Leader notified via mailbox
  → Reassign or create new Teammate
```

## Complete Task Type Overview

Supporting multi-Agent collaboration are 7 task types (`src/tasks/types.ts`):

| Task Type | Execution Location | State Management | Use Case |
|----------|---------|---------|---------|
| **LocalAgentTask** | Local subprocess | `LocalAgentTaskState` | Standard sub-agent tasks |
| **LocalShellTask** | Local shell | `LocalShellTaskState` | Background shell commands |
| **InProcessTeammateTask** | Same process | `InProcessTeammateTaskState` | Lightweight in-process teammates |
| **RemoteAgentTask** | Remote server | `RemoteAgentTaskState` | Distributed Agents (CCR) |
| **DreamTask** | Background silent | `DreamTaskState` | Background autonomous memory organization |
| **LocalWorkflowTask** | Local | `LocalWorkflowTaskState` | Workflow orchestration |
| **MonitorMcpTask** | Local | `MonitorMcpTaskState` | MCP monitoring tasks |

The key difference between `InProcessTeammateTask` and `LocalAgentTask`: the former shares the process's memory space and infrastructure state (such as MCP connection pool) but has independent conversation context and tool permissions; the latter is a fully isolated subprocess with higher startup overhead but better safety.

## Choosing Between Coordinator and Swarm

| Scenario | Recommended Mode | Reason |
|------|---------|------|
| "Refactor the auth system, requires multi-module coordination" | Coordinator | Requires centralized decision-making, Workers have dependencies |
| "Fix 10 independent lint warnings" | Swarm | Tasks are independent, fully parallelizable |
| "Research approach A and approach B, then pick one to implement" | Coordinator | Parallel research first, then centralized decision |
| "Search all TODOs in a large repo and categorize them" | Swarm | No dependencies, each claims tasks independently |
